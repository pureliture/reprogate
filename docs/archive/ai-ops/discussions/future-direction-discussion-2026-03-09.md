## 1. ai-ops를 만들게 된 배경

### 1.1 바이브코딩의 문제점
- 변경 사항에 대한 추적이나 작업 지시 당시의 아이디어가 지속적으로 유실됨
- 문서 갱신, 리팩토링, 오류 검사, 테스트 코드 보완 등 함께 수행되어야 할 작업들을 까먹고 스킵하는 경우 발생
- 컨텍스트가 길어지면 초기에 규정했던 스펙을 까먹어 예상치 못한 결과물이 자주 나옴
- 무작정 바이브코딩하다보면 체계 없이 의식의 흐름대로 기능을 구현하게 되고 나중에는 왜 이 기능을 만들게 됐는지도 까먹음
- 수많은 session에서 의도를 찾을 수 없어서 불필요하게 토큰을 사용하게 됨

### 1.2 사용자의 바이브코딩 사용 패턴
1. **요구사항 구체화**: 추상적 생각을 코드베이스 참고하여 기술적으로 구체화
2. **기능 개발**: 요구사항 구체화 및 계획 검토 후 실제 구현 지시
3. **오류 해결**: 문제 식별 → 원인 분석 + 해결 방법 식별 → 해결(구현)
4. **의심(검증)**: 변경사항이 많거나 요약이 잘 이해되지 않을 때 제대로 이행했는지 검증
5. **문서화**: 지속적으로 문서 갱신 희망, 상황에 따라 갱신해야 할 문서가 다름
6. **리팩토링**: SOLID 원칙, 작은 컴포넌트 + 파사드 패턴, 클린 아키텍처 중시

### 1.3 원하는 것
- 한번의 지시로 원하는 결과물이 바로 나온다고 생각하지 않음
- 작업 규모가 작아도 작업 리뷰 과정은 반드시 필요
- 작업 계획이 뭐였는지, 왜 작업을 계획하게 되었는지(의도), 작업 종료 후 결과물이 어떤 형태를 띄어야 하는지 저장
- 이를 기반으로 지속적으로 평가하는 과정 필요
- 특히 규모가 큰 작업에서는 중간중간 해당 작업을 지속적으로 반복해야 함

## 2. ai-ops의 핵심 비전

### 2.1 본질
```
"내 업무 방식을 AI에게 주입하고,
 주입 후에는 AI가 알아서 따르게 만드는 것"
```

### 2.2 Process as Code
- Infrastructure as Code: 서버 설정을 코드로
- Policy as Code: 보안 정책을 코드로
- **ai-ops: 업무 방식을 코드로 (SoT + Gate)**

한 번 정의하면, 이후로는 자동 적용/강제

### 2.3 바이브코딩 시대의 개발 방법론 포터블 레이어

```
┌─────────────────────────────────────────────────────────┐
│              변하지 않는 것 (Stable)                     │
│  - 개발 방법론 (Agile, TDD, Waterfall...)               │
│  - 팀의 일하는 방식                                      │
│  - 내 스타일                                             │
├─────────────────────────────────────────────────────────┤
│                    ai-ops (추상화 레이어)                │
├─────────────────────────────────────────────────────────┤
│              변하는 것 (Volatile)                        │
│  - AI 도구들 (Claude Code, Codex, Cursor, Copilot...)   │
│  - 기능이 하루가 멀다하고 바뀜                           │
└─────────────────────────────────────────────────────────┘
```

### 2.4 궁극적 목표
- 협업하는 인원들이 또는 개인이 "나는 이런식으로 일했으면 좋겠다"하는 업무의 방향성이나 방식을 ai-ops에 녹일 수 있게 함
- ai-ops에 녹인 이후에는 더 이상 해당 프로세스가 보장됐는지 계속 검증하고, 검증결과 준수 안 했으면 다시 재지시하는 식으로 안 해도 됨
- 바이브코딩 도구를 일종의 한 팀, 프로젝트 TF처럼 써서 해당 팀을 사용자가 "내 팀"처럼 쓸 수 있게 함
- 사람마다 프롬프팅이 다르고 동일한 사람도 날마다 대화하는 방식이 바뀌는데, 이러한 변칙성이 소프트웨어에 녹아나지 않게 함

## 3. ai-ops의 현재 달성 수준

### 3.1 구현된 기능
- 프로세스 정의 (P0-P4, S1-S4, G0)
- WP(Work Packet) 기반 작업 추적
- Hard gate (PreToolUse hook, Git hook)
- 기록 라우팅 (ai-ops vs project-ops)
- 도구 간 SoT 공유 (CLAUDE.md, AGENTS.md)
- Compliance check

### 3.2 세션 비교 결과 (1MB 이상 세션)

**ai-ops 사용 세션**:
- WP/Process 참조 수백 회
- 추적 가능한 구조화된 기록
- 나중에 의도/과정/결과 복원 가능

**ai-ops 미사용 세션**:
- 8.7MB까지 비대해짐
- WP/Process 참조 0
- 추적 불가능한 대화만 존재
- 나중에 복원 불가능

## 4. 비판적 분석

### 4.1 초기 비판 (개인/라이트 사용자 관점)
- 프로세스 선택이 오버헤드
- WP 형식이 너무 무거움
- OMC로 충분하지 않나?
- "과잉 엔지니어링" 가능성

### 4.2 반박 (강제성의 필요성)
- LLM은 확률적 출력 → 같은 입력에도 다른 결과
- 컨텍스트 의존 → 긴 대화에서 초반 지시 망각
- "권장" 무시 → soft reminder는 때때로 skip
- **결론: 강제성 없으면 신뢰 불가능**

### 4.3 팀 표준화 관점에서의 가치
- 누가 작업해도 동일한 프로세스
- 도구 독립적 (Claude/Codex/Gemini 무관)
- 추적 가능성 (WP로 의도/과정/결과 문서화)
- 품질 게이트 (강제 체크포인트로 품질 보장)
- 온보딩 (새 인원도 프로세스 따라가면 됨)

### 4.4 OMC와의 비교

| | OMC | ai-ops |
|---|-----|--------|
| 핵심 가치 | 효율성/생산성 | 프로세스/추적성 |
| 접근 | "자동으로 해줄게" | "이 방식으로 해" |
| 강제성 | 없음 (soft) | 있음 (hard) |
| 즉각적 가치 | 높음 | 낮음 (지연됨) |

OMC = "더 빠르게, 더 많이, 자동으로"
ai-ops = "더 체계적으로, 더 일관되게"

## 5. 오픈소스화 가능성

### 5.1 제안된 구조
```
ai-ops/
├── core/                      # 코어 엔진
│   ├── process-engine/        # 프로세스 실행/검증
│   ├── gate-engine/           # Hard gate 로직
│   └── tracking/              # WP/ADR 추적
│
├── presets/                   # 기본 프로세스 템플릿
│   ├── agile/
│   ├── waterfall/
│   ├── tdd/
│   ├── google-style/
│   ├── amazon-style/
│   └── minimal/
│
├── adapters/                  # AI 도구 어댑터
│   ├── claude/
│   ├── codex/
│   ├── cursor/
│   └── copilot/
│
└── templates/                 # 산출물 템플릿
    ├── wp/
    ├── adr/
    └── checklist/
```

### 5.2 사용자 커스터마이징
```yaml
# .ai-ops/config.yaml
preset: "agile"
overrides:
  phases:
    DEVELOPMENT:
      required_outputs:
        - "code_changes"
        - "tests"
        - "docs_updated"
adapter: "claude"
```

### 5.3 "Process as Code" 오픈소스 현황
- 아직 명확한 경쟁자 없음 (빈 공간)
- "AI 코딩 과정 자체를 체계화하고 강제하는 프레임워크" 부재

## 6. 타겟 사용자 논의

### 6.1 초기 가정: 일반 바이브코딩 사용자
- 문제: 프로세스 강제 거부감, 자유로움 선호
- 결론: 수요 불확실

### 6.2 재정의된 타겟: 미션 크리티컬 시스템 담당자
- 금융/의료/통신/결제 시스템
- 변경 추적 필수, 승인 프로세스 필수
- 감사(audit) 대응, 규제 요구사항
- "AI가 만든 코드를 프로덕션에?" → 프로세스로 신뢰 확보

### 6.3 이 타겟에 대한 비판적 질문 (미검증)
1. 이 타겟이 바이브코딩을 원하긴 하나?
2. ai-ops가 있으면 바이브코딩을 쓸까?
3. 이미 기존 도구(JIRA, CI/CD 등)로 충분하지 않나?
4. 미션 크리티컬 + 바이브코딩 의향 = 교차점이 매우 작을 수 있음
5. 빅테크가 이 영역 진입하면 ai-ops 존재 이유 사라질 수 있음
6. 엔터프라이즈 도입 장벽 (보안/법무/조달/연동/교육)

## 9. 핵심 인사이트 요약

1. **ai-ops의 문제 인식은 정확함**: 바이브코딩의 망각/추적불가/변칙성 문제는 실재
2. **강제성의 필요성은 타당함**: LLM의 확률적 특성상 soft guidance로는 불충분
3. **"도구 독립 방법론 레이어"는 가치 있는 컨셉**: AI 도구는 바뀌지만 방법론은 축적됨
4. **하지만 수요는 검증 안 됨**: 실제 사용자가 원하는지, 시장이 있는지 불확실
5. **타겟 명확화 필요**: 라이트 사용자 vs 미션 크리티컬 담당자 → 가치 제안이 완전히 다름

## 10. 후속 논의 (2026-03-10)

### 10.1 비판적 도전과 검증

#### 제기된 비판

1. **"강제성이 필요하다"는 논리의 함정**
   - Hard gate는 "잘못된 행동을 막는 것"이지 "올바른 행동을 유도"하는 게 아님
   - Compliance check가 "형식적 준수"만 검증하고 "실질적 준수"는 검증 못 함

2. **WP 기반 추적의 실제 가치 의문**
   - WP 참조 횟수가 높으면 뭐가 좋은가?
   - 실제로 복원해서 쓴 적 있는가?
   - Git blame + commit message로 충분하지 않은가?

3. **"Process as Code" 비유의 한계**
   - IaC: 코드 = 실제 상태 (결정론적)
   - ai-ops: 코드 ≠ 실제 행동 (확률적 LLM)

#### 사용자 반박 (경험적 증거)

1. **Git commit message 현실**
   - 글로벌 프로젝트에서 커밋 메시지 품질이 들쭉날쭉
   - "API 연동 기능 추가" 수준의 메시지가 많음
   - 대규모 커밋에서 AI가 변경사항의 1/3만 커버하는 경우 발생

2. **Compaction 생존 = 핵심 가치**
   - Compact 이후에도 WP를 찾아가서 마치 compact가 일어나지 않은 것처럼 대화가 매끄러웠음
   - WP = 외부 메모리로서 컨텍스트 한계 우회

3. **생산성 4배 증가**
   - 불필요한 티키타카 감소
   - 요구사항 누락 후 재지시 → compaction → 맥락 손실 사이클이 끊김

4. **Soft guidance 실패 경험**
   - Soft guidance만 적용했더니 매번 대화마다 불안정하게 동작
   - ADR 안 남기고 대뜸 구현하거나, user_guide.md 최신화 안 하고 커밋하려 함
   - 이 경험 때문에 hard gate를 계속 적용하게 됨

5. **부수 작업 누락의 누적 문제**
   - LLM이 "이건 user_guide 갱신이 필요해"라고 스스로 판단하길 기대하는 건 도박
   - 누락이 누적되면 문서와 실제 기능이 완전히 따로 놀게 됨

### 10.2 검증된 ai-ops 핵심 가치

| 구성요소 | 역할 | 없으면? |
|---------|------|--------|
| **WP** | 외부 메모리, 맥락 보존 | Compaction 후 맥락 손실 |
| **프로세스 정의** | 실행 패턴 규격화 | 매번 다른 방식으로 실행, 사용자가 매번 검증 필요 |
| **Hard Gate** | 부수 작업 강제 | 문서/ADR/테스트 누락 누적 |

**결론**: 세 개가 다 필요함. WP만으로도 안 되고, 프로세스 정의만으로도 안 되고, Gate까지 있어야 완성.

### 10.3 목표 재정의

> "거창하게 OMC급 오픈소스가 되는 걸 바라는 건 아니야. 회사 내부에서라도 확산됐으면... 내부 인원들이 사용을 시도해보고 또 사용하고 싶어질 수준 정도"

- 타겟: 외부 오픈소스 커뮤니티 → **회사 내부 동료**
- 목표: 대규모 확산 → **N=2~5 검증 및 점진적 확산**
- 성공 기준: 동료가 "써보고 다시 쓰고 싶어지는" 수준

### 10.4 내부 확산의 장벽

1. **진입 장벽이 높음**
   - 프로세스 카탈로그 (P0-P4, S1-S4, G0)
   - WP 형식
   - Hard gate 설정
   - 여러 문서 참조 필요
   - 동료 입장: "문서만 10개네. 그냥 Claude Code 쓸래."

2. **즉각적 가치가 안 보임**
   - 가치 체감은 수주~수개월 사용 후
   - 첫 사용에서 "왜 이게 좋은지" 안 보임

3. **문제 인식이 공유되지 않음**
   - 동료들이 같은 pain point를 경험했는지 불확실
   - 문제를 인식조차 못 했을 수도 있음

### 10.5 내부 확산 전략 (제안)

#### Phase 1: Before/After 스토리 공유

코드나 프레임워크 전에 스토리를 먼저 공유:

```
[Before]
- 기능 10개 추가했는데 왜 추가했는지 기억 안 남
- Compaction 후 "뭐 하고 있었지?" 반복
- user_guide.md 3개월째 방치

[After - ai-ops 적용]
- WP에 의도/결과 다 남아있음
- Compaction 후에도 WP 보고 이어감
- Hard gate가 문서 갱신 강제
```

**목표**: "아 나도 저 문제 있는데?" 공감 유도

#### Phase 2: ai-ops-lite 제공

전체 ai-ops가 아니라 최소 버전:

```
ai-ops-lite/
├── CLAUDE.md (또는 AGENTS.md)  # 핵심 지시문만
├── templates/
│   └── wp-simple.md            # WP 템플릿 1개
├── processes/
│   ├── specify.md              # 요구사항 구체화
│   ├── implement.md            # 구현
│   └── verify.md               # 검증
└── hooks/
    └── pre-tool-use.sh         # 최소 gate
```

프로세스 3개, WP 템플릿 1개, Hook 1개. **이게 전부**.

#### Phase 3: 한 명씩 온보딩

1. 관심 있는 동료 1명 찾기
2. 페어로 같이 써보기
3. 그 사람 피드백으로 개선
4. 그 사람이 다음 사람 온보딩

**목표**: N=2 검증 → N=3 → N=5

### 10.6 "버려야 하나?"에 대한 결론

**개인 사용**: 절대 버리면 안 됨
- 이미 작동하고 있음
- ROI가 증명됨 (생산성 4배, 티키타카 감소, compaction 생존)

**내부 확산**: 검증이 필요할 뿐
- N=1 검증 완료 (본인)
- N>1 검증 미완료 (동료들)

**핵심**: ai-ops는 "만든 사람한테 작동하는 도구" = **1차 성공**. 2차 성공(다른 사람에게도 작동)은 별개의 검증 필요.

## 12. 3자 토론 결과 (2026-03-10)

> 세 페르소나(옹호론자 A, 비판론자 B, 중재자 C)를 통한 구조화된 토론 수행

### 12.1 옹호론자(A)의 핵심 주장

1. **LLM의 구조적 결함은 도구 성능으로 해결 안 됨**
   - 망각(컨텍스트 한계), 변칙성(확률적 입출력), 누락의 누적
   - 모델이 좋아져도 이 문제들은 사라지지 않음

2. **"Process as Code"는 자연스러운 진화**
   - IaC(서버) → PaC(정책) → 이제 프로세스 차례
   - 한 번 정의하면 자동 적용/강제

3. **시장에 빈 공간 존재**
   - "AI 에이전트의 작업 프로세스를 강제하는 프레임워크" 부재
   - 빅테크는 자사 도구 중심 → 도구 독립 레이어는 커뮤니티에서

4. **이미 검증됨**
   - N=1 검증 완료 (생산성 4배, compaction 생존)
   - 포터블 아키텍처 실증 (Claude+OMC / Codex+OMX 이중 어댑터)

### 12.2 비판론자(B)의 핵심 주장

1. **추상화 레이어는 허상**
   - 방법론도 변함 (Waterfall → Agile → Lean...)
   - 도구 차이가 너무 커서 어댑터 ≠ 진정한 추상화

2. **"Process as Code" 비유의 결함**
   - IaC: 코드 = 실제 상태 (결정론적)
   - ai-ops: 코드 ≠ 실제 행동 (확률적 LLM)
   - Gate는 형식적 준수만 검증 가능

3. **기존 도구로 충분**
   - JIRA + GitHub Actions + CLAUDE.md 3줄
   - 80%의 편익을 5%의 비용으로

4. **ROI 정당화 어려움**
   - N=1은 생존자 편향
   - 정량화 불가능한 편익 vs 측정 가능한 비용
   - 프레임워크 구축 비용 회수 불확실

### 12.3 양측 공통점

| 영역 | 합의 |
|------|------|
| 문제 인식 | 바이브코딩의 맥락 유실, 부수 작업 누락, 추적 불가능성은 실재 |
| 외부 메모리 가치 | 외부에 맥락을 기록하는 것 자체는 가치 있음 |
| 강제 필요성 | 어떤 형태로든 강제 메커니즘이 필요 |

### 12.4 핵심 쟁점

1. **복잡도 vs 가치**: 10개 프로세스가 필요한가, CLAUDE.md 3줄로 충분한가?
2. **포터블리티 실현 가능성**: SoT + 어댑터가 진짜 유지보수 비용을 줄이는가?
3. **Gate 위치**: LLM 내부(PreToolUse) vs 외부(CI/CD) vs 둘 다?
4. **ROI 측정**: 정성적 편익을 어떻게 평가할 것인가?

### 12.5 중재 결론

**ai-ops를 버려야 하는가?** → **아니오**

**현재 형태로 유지해야 하는가?** → **아니오**

**합의점**:
> ai-ops의 **핵심 가치(외부 메모리 + 프로세스 정의 + 강제 메커니즘)**는 유효하다.
> 다만 **현재 형태는 과도하게 복잡**하며, **ai-ops-lite로 단순화**하여 N>1 검증을 먼저 수행해야 한다.
> 포터블리티는 장기 비전으로 유지하되, 당장은 **Claude 생태계에 집중**한다.

### 12.6 합의된 다음 단계

| 즉시 실행 | 보류 (검증 후 결정) |
|-----------|---------------------|
| ai-ops-lite 정의 (프로세스 3개, WP 1개, Hook 1개) | 프로세스 확장 (P0-P4, S1-S4) |
| 외부 Gate 추가 (GitHub Actions) | 어댑터 추가 (Cursor, Copilot) |
| 첫 테스트 유저 확보 및 피드백 수집 | 오픈소스화 |

## 13. 제품화 방향 논의 (2026-03-10)

### 13.1 필수 개념 재정립 (6개)

| # | 개념 | 정의 | 목적 |
|---|------|------|------|
| 1 | **외부 메모리** | 세션/도구 전환 시 맥락 유지 | 세션 생존 |
| 2 | **프로세스 정의** | 작업 유형별 절차 명시 | 일관된 실행 |
| 3 | **강제 메커니즘** | LLM이 절차를 반드시 따르게 함 | 준수 보장 |
| 4 | **추적 가능성** | 왜/뭘 했는지 복원 가능 | 과거 복원 |
| 5 | **도구 독립성** | 특정 AI 도구에 종속 안 됨 | 벤더 종속 방지 |
| 6 | **통합 준비성** | 기록이 구조화되어 외부 도구에 연결 가능 | 미래 확장 |

### 13.2 통합 준비성의 의미

> "기록을 남기는 이유는 여러가지 확장에 사용될 수 있는 '재료', '밑바탕'을 남기는 개념"

- MCP, Notion, Confluence, 외부 훅, 파이프라인 등을 나중에 붙일 때
- 기록이 구조화되어 있지 않으면 **프로젝트를 다시 시작한 격**이 됨
- 구조화된 기록(YAML/JSON)은 **파싱 가능한 밑바탕**

### 13.3 최소 제품 형태: CLI

```
┌─────────────────────────────────────────────┐
│              ai-ops CLI                      │
│  $ ai-ops start "작업 제목"                  │
│  $ ai-ops check                              │
│  $ ai-ops done                               │
└─────────────────────┬───────────────────────┘
                      │
                      ▼
┌─────────────────────────────────────────────┐
│           Repository (SoT)                   │
│  .ai-ops/                                    │
│  ├── work-logs/     ← 구조화된 기록 (YAML)   │
│  ├── config.yaml    ← 프로세스 정의          │
│  └── schema.json    ← 기록 스키마            │
└─────────────────────┬───────────────────────┘
                      │
        (나중에 붙임)  │
        ┌─────────────┼─────────────┐
        ▼             ▼             ▼
     MCP Server    Notion Sync   Pipeline Hook
```

### 13.4 포지셔닝: 팀 표준화 (B)

> "팀 전체가 동일한 AI 협업 프로세스를 따르고, 일관된 기록을 남기며, 나중에 어떤 도구든 붙일 수 있는 밑바탕을 만드는 CLI"

**팀 표준화의 조건:**
1. 같은 CLI → 같은 명령어, 같은 워크플로우
2. 같은 기록 형식 → 누가 작업해도 같은 스키마
3. 같은 repo 구조 → `.ai-ops/` 디렉토리 표준

### 13.5 필수 개념 vs 구현 방식 (재정리)

| 필수 개념 (불변) | 구현 방식 (가변) |
|-----------------|-----------------|
| 외부 메모리 필요 | WP? 자유 텍스트? YAML? |
| 프로세스 정의 필요 | P0-P4? 3개? Amazon 6-pager? |
| 강제 메커니즘 필요 | Hook? CI/CD? CLI? |
| 추적 가능성 필요 | Git? 별도 DB? |
| 도구 독립성 필요 | 어댑터? SoT 중심? |
| 통합 준비성 필요 | JSON? YAML? GraphQL? |

### 13.6 빅테크 프로세스/템플릿 공개 현황 조사

#### 공식 공개된 자료

| 회사 | 공개 자료 | 내용 | 라이선스 |
|------|----------|------|----------|
| **Microsoft** | [Code-with Engineering Playbook](https://github.com/microsoft/code-with-engineering-playbook) | Agile, Code Review, CI/CD, Design, Documentation, Security 등 15개 영역 가이드 + Engineering Fundamentals Checklist | MIT |
| **Google** | [Engineering Practices](https://google.github.io/eng-practices/) | Code Review 가이드 (Reviewer's Guide + Change Author's Guide) | CC-By 3.0 |
| **커뮤니티** | [ADR Templates](https://github.com/joelparkerhenderson/architecture-decision-record) | 12개 이상의 ADR 템플릿 (Michael Nygard, MADR, arc42 등) | 다양함 |

#### 비공식/부분 공개

| 회사 | 자료 | 상태 |
|------|------|------|
| **Amazon** | 6-pager, PR/FAQ, Working Backwards | 공식 템플릿 없음. 책/블로그에서 설명만 |
| **Meta** | 내부 프로세스 | 공개 안 됨 |
| **Stripe** | RFC 프로세스 | 블로그 언급만, 템플릿 없음 |

#### 프리셋 전략

**안전하게 사용 가능 (공식 오픈소스)**:
```
presets/
├── minimal/           # 최소 (start, check, done만)
├── standard/          # 표준 (분석, 구현, 검증, 문서화)
├── microsoft-ise/     # Microsoft Engineering Fundamentals 기반 (MIT)
├── google-review/     # Google Code Review 가이드 기반 (CC-By 3.0)
├── adr-nygard/        # Michael Nygard ADR 템플릿
├── adr-madr/          # MADR 템플릿 (MIT)
└── custom/            # 사용자 정의
```

**영감만 받을 것 (직접 인용 X)**:

| 개념 | 영감 출처 | 우리 이름 |
|------|----------|----------|
| 6-pager | Amazon | `narrative-design-doc` |
| PR/FAQ | Amazon | `press-release-driven` |
| RFC | Stripe/Rust | `rfc-driven` |

#### 결론

1. **Microsoft, Google 자료는 바로 사용 가능** - 라이선스 명확
2. **Amazon, Meta 등은 공식 템플릿 없음** - "inspired by" 형태로만
3. **ADR 템플릿은 커뮤니티에서 풍부하게 제공**
4. **프리셋으로 제공 + 커스텀 지원** 구조가 현실적
