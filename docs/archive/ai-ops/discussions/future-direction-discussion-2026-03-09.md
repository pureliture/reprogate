## 1. ai-ops를 만들게 된 배경

### 1.1 바이브코딩의 문제점
- 변경 사항에 대한 추적이나 작업 지시 당시의 아이디어가 지속적으로 유실됨
- 문서 갱신, 리팩토링, 오류 검사, 테스트 코드 보완 등 함께 수행되어야 할 작업들을 까먹고 스킵하는 경우 발생
- 컨텍스트가 길어지면 초기에 규정했던 스펙을 까먹어 예상치 못한 결과물이 자주 나옴
- 무작정 바이브코딩하다보면 체계 없이 의식의 흐름대로 기능을 구현하게 되고 나중에는 왜 이 기능을 만들게 됐는지도 까먹음
- 수많은 session에서 의도를 찾을 수 없어서 불필요하게 토큰을 사용하게 됨

### 1.2 사용자의 바이브코딩 사용 패턴
1. **요구사항 구체화**: 추상적 생각을 코드베이스 참고하여 기술적으로 구체화
2. **기능 개발**: 요구사항 구체화 및 계획 검토 후 실제 구현 지시
3. **오류 해결**: 문제 식별 → 원인 분석 + 해결 방법 식별 → 해결(구현)
4. **의심(검증)**: 변경사항이 많거나 요약이 잘 이해되지 않을 때 제대로 이행했는지 검증
5. **문서화**: 지속적으로 문서 갱신 희망, 상황에 따라 갱신해야 할 문서가 다름
6. **리팩토링**: SOLID 원칙, 작은 컴포넌트 + 파사드 패턴, 클린 아키텍처 중시

### 1.3 원하는 것
- 한번의 지시로 원하는 결과물이 바로 나온다고 생각하지 않음
- 작업 규모가 작아도 작업 리뷰 과정은 반드시 필요
- 작업 계획이 뭐였는지, 왜 작업을 계획하게 되었는지(의도), 작업 종료 후 결과물이 어떤 형태를 띄어야 하는지 저장
- 이를 기반으로 지속적으로 평가하는 과정 필요
- 특히 규모가 큰 작업에서는 중간중간 해당 작업을 지속적으로 반복해야 함

## 2. ai-ops의 핵심 비전

### 2.1 본질
```
"내 업무 방식을 AI에게 주입하고,
 주입 후에는 AI가 알아서 따르게 만드는 것"
```

### 2.2 Process as Code
- Infrastructure as Code: 서버 설정을 코드로
- Policy as Code: 보안 정책을 코드로
- **ai-ops: 업무 방식을 코드로 (SoT + Gate)**

한 번 정의하면, 이후로는 자동 적용/강제

### 2.3 바이브코딩 시대의 개발 방법론 포터블 레이어

```
┌─────────────────────────────────────────────────────────┐
│              변하지 않는 것 (Stable)                     │
│  - 개발 방법론 (Agile, TDD, Waterfall...)               │
│  - 팀의 일하는 방식                                      │
│  - 내 스타일                                             │
├─────────────────────────────────────────────────────────┤
│                    ai-ops (추상화 레이어)                │
├─────────────────────────────────────────────────────────┤
│              변하는 것 (Volatile)                        │
│  - AI 도구들 (Claude Code, Codex, Cursor, Copilot...)   │
│  - 기능이 하루가 멀다하고 바뀜                           │
└─────────────────────────────────────────────────────────┘
```

### 2.4 궁극적 목표
- 협업하는 인원들이 또는 개인이 "나는 이런식으로 일했으면 좋겠다"하는 업무의 방향성이나 방식을 ai-ops에 녹일 수 있게 함
- ai-ops에 녹인 이후에는 더 이상 해당 프로세스가 보장됐는지 계속 검증하고, 검증결과 준수 안 했으면 다시 재지시하는 식으로 안 해도 됨
- 바이브코딩 도구를 일종의 한 팀, 프로젝트 TF처럼 써서 해당 팀을 사용자가 "내 팀"처럼 쓸 수 있게 함
- 사람마다 프롬프팅이 다르고 동일한 사람도 날마다 대화하는 방식이 바뀌는데, 이러한 변칙성이 소프트웨어에 녹아나지 않게 함

## 3. ai-ops의 현재 달성 수준

### 3.1 구현된 기능
- 프로세스 정의 (P0-P4, S1-S4, G0)
- WP(Work Packet) 기반 작업 추적
- Hard gate (PreToolUse hook, Git hook)
- 기록 라우팅 (ai-ops vs project-ops)
- 도구 간 SoT 공유 (CLAUDE.md, AGENTS.md)
- Compliance check

### 3.2 세션 비교 결과 (1MB 이상 세션)

**ai-ops 사용 세션**:
- WP/Process 참조 수백 회
- 추적 가능한 구조화된 기록
- 나중에 의도/과정/결과 복원 가능

**ai-ops 미사용 세션**:
- 8.7MB까지 비대해짐
- WP/Process 참조 0
- 추적 불가능한 대화만 존재
- 나중에 복원 불가능

## 4. 비판적 분석

### 4.1 초기 비판 (개인/라이트 사용자 관점)
- 프로세스 선택이 오버헤드
- WP 형식이 너무 무거움
- OMC로 충분하지 않나?
- "과잉 엔지니어링" 가능성

### 4.2 반박 (강제성의 필요성)
- LLM은 확률적 출력 → 같은 입력에도 다른 결과
- 컨텍스트 의존 → 긴 대화에서 초반 지시 망각
- "권장" 무시 → soft reminder는 때때로 skip
- **결론: 강제성 없으면 신뢰 불가능**

### 4.3 팀 표준화 관점에서의 가치
- 누가 작업해도 동일한 프로세스
- 도구 독립적 (Claude/Codex/Gemini 무관)
- 추적 가능성 (WP로 의도/과정/결과 문서화)
- 품질 게이트 (강제 체크포인트로 품질 보장)
- 온보딩 (새 인원도 프로세스 따라가면 됨)

### 4.4 OMC와의 비교

| | OMC | ai-ops |
|---|-----|--------|
| 핵심 가치 | 효율성/생산성 | 프로세스/추적성 |
| 접근 | "자동으로 해줄게" | "이 방식으로 해" |
| 강제성 | 없음 (soft) | 있음 (hard) |
| 즉각적 가치 | 높음 | 낮음 (지연됨) |

OMC = "더 빠르게, 더 많이, 자동으로"
ai-ops = "더 체계적으로, 더 일관되게"

## 5. 오픈소스화 가능성

### 5.1 제안된 구조
```
ai-ops/
├── core/                      # 코어 엔진
│   ├── process-engine/        # 프로세스 실행/검증
│   ├── gate-engine/           # Hard gate 로직
│   └── tracking/              # WP/ADR 추적
│
├── presets/                   # 기본 프로세스 템플릿
│   ├── agile/
│   ├── waterfall/
│   ├── tdd/
│   ├── google-style/
│   ├── amazon-style/
│   └── minimal/
│
├── adapters/                  # AI 도구 어댑터
│   ├── claude/
│   ├── codex/
│   ├── cursor/
│   └── copilot/
│
└── templates/                 # 산출물 템플릿
    ├── wp/
    ├── adr/
    └── checklist/
```

### 5.2 사용자 커스터마이징
```yaml
# .ai-ops/config.yaml
preset: "agile"
overrides:
  phases:
    DEVELOPMENT:
      required_outputs:
        - "code_changes"
        - "tests"
        - "docs_updated"
adapter: "claude"
```

### 5.3 "Process as Code" 오픈소스 현황
- 아직 명확한 경쟁자 없음 (빈 공간)
- "AI 코딩 과정 자체를 체계화하고 강제하는 프레임워크" 부재

## 6. 타겟 사용자 논의

### 6.1 초기 가정: 일반 바이브코딩 사용자
- 문제: 프로세스 강제 거부감, 자유로움 선호
- 결론: 수요 불확실

### 6.2 재정의된 타겟: 미션 크리티컬 시스템 담당자
- 금융/의료/통신/결제 시스템
- 변경 추적 필수, 승인 프로세스 필수
- 감사(audit) 대응, 규제 요구사항
- "AI가 만든 코드를 프로덕션에?" → 프로세스로 신뢰 확보

### 6.3 이 타겟에 대한 비판적 질문 (미검증)
1. 이 타겟이 바이브코딩을 원하긴 하나?
2. ai-ops가 있으면 바이브코딩을 쓸까?
3. 이미 기존 도구(JIRA, CI/CD 등)로 충분하지 않나?
4. 미션 크리티컬 + 바이브코딩 의향 = 교차점이 매우 작을 수 있음
5. 빅테크가 이 영역 진입하면 ai-ops 존재 이유 사라질 수 있음
6. 엔터프라이즈 도입 장벽 (보안/법무/조달/연동/교육)

## 9. 핵심 인사이트 요약

1. **ai-ops의 문제 인식은 정확함**: 바이브코딩의 망각/추적불가/변칙성 문제는 실재
2. **강제성의 필요성은 타당함**: LLM의 확률적 특성상 soft guidance로는 불충분
3. **"도구 독립 방법론 레이어"는 가치 있는 컨셉**: AI 도구는 바뀌지만 방법론은 축적됨
4. **하지만 수요는 검증 안 됨**: 실제 사용자가 원하는지, 시장이 있는지 불확실
5. **타겟 명확화 필요**: 라이트 사용자 vs 미션 크리티컬 담당자 → 가치 제안이 완전히 다름

## 10. 후속 논의 (2026-03-10)

### 10.1 비판적 도전과 검증

#### 제기된 비판

1. **"강제성이 필요하다"는 논리의 함정**
   - Hard gate는 "잘못된 행동을 막는 것"이지 "올바른 행동을 유도"하는 게 아님
   - Compliance check가 "형식적 준수"만 검증하고 "실질적 준수"는 검증 못 함

2. **WP 기반 추적의 실제 가치 의문**
   - WP 참조 횟수가 높으면 뭐가 좋은가?
   - 실제로 복원해서 쓴 적 있는가?
   - Git blame + commit message로 충분하지 않은가?

3. **"Process as Code" 비유의 한계**
   - IaC: 코드 = 실제 상태 (결정론적)
   - ai-ops: 코드 ≠ 실제 행동 (확률적 LLM)

#### 사용자 반박 (경험적 증거)

1. **Git commit message 현실**
   - 글로벌 프로젝트에서 커밋 메시지 품질이 들쭉날쭉
   - "API 연동 기능 추가" 수준의 메시지가 많음
   - 대규모 커밋에서 AI가 변경사항의 1/3만 커버하는 경우 발생

2. **Compaction 생존 = 핵심 가치**
   - Compact 이후에도 WP를 찾아가서 마치 compact가 일어나지 않은 것처럼 대화가 매끄러웠음
   - WP = 외부 메모리로서 컨텍스트 한계 우회

3. **생산성 4배 증가**
   - 불필요한 티키타카 감소
   - 요구사항 누락 후 재지시 → compaction → 맥락 손실 사이클이 끊김

4. **Soft guidance 실패 경험**
   - Soft guidance만 적용했더니 매번 대화마다 불안정하게 동작
   - ADR 안 남기고 대뜸 구현하거나, user_guide.md 최신화 안 하고 커밋하려 함
   - 이 경험 때문에 hard gate를 계속 적용하게 됨

5. **부수 작업 누락의 누적 문제**
   - LLM이 "이건 user_guide 갱신이 필요해"라고 스스로 판단하길 기대하는 건 도박
   - 누락이 누적되면 문서와 실제 기능이 완전히 따로 놀게 됨

### 10.2 검증된 ai-ops 핵심 가치

| 구성요소 | 역할 | 없으면? |
|---------|------|--------|
| **WP** | 외부 메모리, 맥락 보존 | Compaction 후 맥락 손실 |
| **프로세스 정의** | 실행 패턴 규격화 | 매번 다른 방식으로 실행, 사용자가 매번 검증 필요 |
| **Hard Gate** | 부수 작업 강제 | 문서/ADR/테스트 누락 누적 |

**결론**: 세 개가 다 필요함. WP만으로도 안 되고, 프로세스 정의만으로도 안 되고, Gate까지 있어야 완성.

### 10.3 목표 재정의

> "거창하게 OMC급 오픈소스가 되는 걸 바라는 건 아니야. 회사 내부에서라도 확산됐으면... 내부 인원들이 사용을 시도해보고 또 사용하고 싶어질 수준 정도"

- 타겟: 외부 오픈소스 커뮤니티 → **회사 내부 동료**
- 목표: 대규모 확산 → **N=2~5 검증 및 점진적 확산**
- 성공 기준: 동료가 "써보고 다시 쓰고 싶어지는" 수준

### 10.4 내부 확산의 장벽

1. **진입 장벽이 높음**
   - 프로세스 카탈로그 (P0-P4, S1-S4, G0)
   - WP 형식
   - Hard gate 설정
   - 여러 문서 참조 필요
   - 동료 입장: "문서만 10개네. 그냥 Claude Code 쓸래."

2. **즉각적 가치가 안 보임**
   - 가치 체감은 수주~수개월 사용 후
   - 첫 사용에서 "왜 이게 좋은지" 안 보임

3. **문제 인식이 공유되지 않음**
   - 동료들이 같은 pain point를 경험했는지 불확실
   - 문제를 인식조차 못 했을 수도 있음

### 10.5 내부 확산 전략 (제안)

#### Phase 1: Before/After 스토리 공유

코드나 프레임워크 전에 스토리를 먼저 공유:

```
[Before]
- 기능 10개 추가했는데 왜 추가했는지 기억 안 남
- Compaction 후 "뭐 하고 있었지?" 반복
- user_guide.md 3개월째 방치

[After - ai-ops 적용]
- WP에 의도/결과 다 남아있음
- Compaction 후에도 WP 보고 이어감
- Hard gate가 문서 갱신 강제
```

**목표**: "아 나도 저 문제 있는데?" 공감 유도

#### Phase 2: ai-ops-lite 제공

전체 ai-ops가 아니라 최소 버전:

```
ai-ops-lite/
├── CLAUDE.md (또는 AGENTS.md)  # 핵심 지시문만
├── templates/
│   └── wp-simple.md            # WP 템플릿 1개
├── processes/
│   ├── specify.md              # 요구사항 구체화
│   ├── implement.md            # 구현
│   └── verify.md               # 검증
└── hooks/
    └── pre-tool-use.sh         # 최소 gate
```

프로세스 3개, WP 템플릿 1개, Hook 1개. **이게 전부**.

#### Phase 3: 한 명씩 온보딩

1. 관심 있는 동료 1명 찾기
2. 페어로 같이 써보기
3. 그 사람 피드백으로 개선
4. 그 사람이 다음 사람 온보딩

**목표**: N=2 검증 → N=3 → N=5

### 10.6 "버려야 하나?"에 대한 결론

**개인 사용**: 절대 버리면 안 됨
- 이미 작동하고 있음
- ROI가 증명됨 (생산성 4배, 티키타카 감소, compaction 생존)

**내부 확산**: 검증이 필요할 뿐
- N=1 검증 완료 (본인)
- N>1 검증 미완료 (동료들)

**핵심**: ai-ops는 "만든 사람한테 작동하는 도구" = **1차 성공**. 2차 성공(다른 사람에게도 작동)은 별개의 검증 필요.

## 12. 3자 토론 결과 (2026-03-10)

> 세 페르소나(옹호론자 A, 비판론자 B, 중재자 C)를 통한 구조화된 토론 수행

### 12.1 옹호론자(A)의 핵심 주장

1. **LLM의 구조적 결함은 도구 성능으로 해결 안 됨**
   - 망각(컨텍스트 한계), 변칙성(확률적 입출력), 누락의 누적
   - 모델이 좋아져도 이 문제들은 사라지지 않음

2. **"Process as Code"는 자연스러운 진화**
   - IaC(서버) → PaC(정책) → 이제 프로세스 차례
   - 한 번 정의하면 자동 적용/강제

3. **시장에 빈 공간 존재**
   - "AI 에이전트의 작업 프로세스를 강제하는 프레임워크" 부재
   - 빅테크는 자사 도구 중심 → 도구 독립 레이어는 커뮤니티에서

4. **이미 검증됨**
   - N=1 검증 완료 (생산성 4배, compaction 생존)
   - 포터블 아키텍처 실증 (Claude+OMC / Codex+OMX 이중 어댑터)

### 12.2 비판론자(B)의 핵심 주장

1. **추상화 레이어는 허상**
   - 방법론도 변함 (Waterfall → Agile → Lean...)
   - 도구 차이가 너무 커서 어댑터 ≠ 진정한 추상화

2. **"Process as Code" 비유의 결함**
   - IaC: 코드 = 실제 상태 (결정론적)
   - ai-ops: 코드 ≠ 실제 행동 (확률적 LLM)
   - Gate는 형식적 준수만 검증 가능

3. **기존 도구로 충분**
   - JIRA + GitHub Actions + CLAUDE.md 3줄
   - 80%의 편익을 5%의 비용으로

4. **ROI 정당화 어려움**
   - N=1은 생존자 편향
   - 정량화 불가능한 편익 vs 측정 가능한 비용
   - 프레임워크 구축 비용 회수 불확실

### 12.3 양측 공통점

| 영역 | 합의 |
|------|------|
| 문제 인식 | 바이브코딩의 맥락 유실, 부수 작업 누락, 추적 불가능성은 실재 |
| 외부 메모리 가치 | 외부에 맥락을 기록하는 것 자체는 가치 있음 |
| 강제 필요성 | 어떤 형태로든 강제 메커니즘이 필요 |

### 12.4 핵심 쟁점

1. **복잡도 vs 가치**: 10개 프로세스가 필요한가, CLAUDE.md 3줄로 충분한가?
2. **포터블리티 실현 가능성**: SoT + 어댑터가 진짜 유지보수 비용을 줄이는가?
3. **Gate 위치**: LLM 내부(PreToolUse) vs 외부(CI/CD) vs 둘 다?
4. **ROI 측정**: 정성적 편익을 어떻게 평가할 것인가?

### 12.5 중재 결론

**ai-ops를 버려야 하는가?** → **아니오**

**현재 형태로 유지해야 하는가?** → **아니오**

**합의점**:
> ai-ops의 **핵심 가치(외부 메모리 + 프로세스 정의 + 강제 메커니즘)**는 유효하다.
> 다만 **현재 형태는 과도하게 복잡**하며, **ai-ops-lite로 단순화**하여 N>1 검증을 먼저 수행해야 한다.
> 포터블리티는 장기 비전으로 유지하되, 당장은 **Claude 생태계에 집중**한다.

### 12.6 합의된 다음 단계

| 즉시 실행 | 보류 (검증 후 결정) |
|-----------|---------------------|
| ai-ops-lite 정의 (프로세스 3개, WP 1개, Hook 1개) | 프로세스 확장 (P0-P4, S1-S4) |

## 14. Agent Chaining과의 스코프 차이 (2026-03-12)

### 14.1 제기된 질문

> "ai-ops를 현재 방식 말고 agent chaining으로 구성하는 게 더 나은 방법 아니었을까?"
> "각 프로세스들을 에이전트들이 수행하는 방식이 지금 방식보다 더 하드하게 보장하는 방법 아닌가?"

### 14.2 결론: 스코프가 다르다

**Agent chaining과 ai-ops는 경쟁 관계가 아니라 다른 레이어에서 작동한다.**

| | Agent Chaining | ai-ops |
|---|---|---|
| **초점** | 작업 흐름 (workflow) | 운영 규율 / 강제 (governance) |
| **관심사** | "요구사항→설계→테스트→리뷰를 **누가 어떤 순서로** 수행하나" | "각 단계에서 에이전트가 **무엇을 반드시** 해야 하나" |
| **단위** | 기능 개발 단계 | 단계 내 동작 규칙 |
| **목적** | 실행 효율, 역할 분리, 전문화 | 준수 보장, 기록 강제, 추적 가능성 |
| **비유** | 파이프라인 설계 ("누가 → 누구에게") | 각 노드의 행동 규범 ("이렇게 해야 함") |

### 14.3 관계 정리

```
┌─────────────────────────────────────────────────────────────┐
│                    Agent Chaining Layer                      │
│         (요구사항 Agent → 설계 Agent → 구현 Agent → ...)      │
├─────────────────────────────────────────────────────────────┤
│                      ai-ops Layer                            │
│   각 Agent가 실행될 때:                                       │
│   - 어떤 프로세스를 따라야 하는가 (P0-P4)                      │
│   - 어떤 기록을 남겨야 하는가 (WP, ADR)                        │
│   - 어떤 Gate를 통과해야 하는가 (Hard Gate)                    │
└─────────────────────────────────────────────────────────────┘
```

### 14.4 "더 하드하게 보장" 주장에 대한 분석

**맞는 부분:**
- Agent chaining으로 `P0 → P1 → P3` 흐름을 강제하면 순서 보장이 더 명확해짐
- `executor → verifier → recorder` 분리가 실제 실행 단위가 됨
- 사용자가 중간에 임의로 단계를 스킵하기 어려움

**틀린 부분:**
- Agent chaining은 **흐름만 강제**하지, **각 단계에서 무엇을 해야 하는지**는 강제하지 않음
- 예: "설계 Agent"가 실행됐다고 해서 ADR을 남겼는지, 문서를 갱신했는지는 보장 안 됨
- ai-ops의 Gate가 바로 이 부분을 담당

### 14.5 결론

1. **Agent chaining은 ai-ops를 대체하지 않는다** — 다른 문제를 해결함
2. **상호 보완적 관계** — Agent chaining 위에 ai-ops가 올라가거나, ai-ops 프로세스를 agent chain으로 실행 가능
3. **ai-ops의 핵심 가치는 유지** — 외부 메모리 + 프로세스 정의 + 강제 메커니즘
4. **향후 고려사항** — Agent chaining을 ai-ops의 실행 레이어로 도입하는 것은 검토 가능하나, 현재 우선순위는 아님

## 15. 포지셔닝 확정 (2026-03-12)

### 15.1 슬로건 (확정)

> **"ai-ops는 개발 방법론을 LLM에게 이식해, LLM이 당신처럼 일하게 만든다"**

### 15.2 핵심 포지셔닝: 레이어 구분

```
┌─────────────────────────────────────────────────────────────┐
│  팀/시스템 레벨                                              │
│  CI/CD, 코드 리뷰, 린터                                      │
│  → "결과물이 기준을 통과하나?"                                │
├─────────────────────────────────────────────────────────────┤
│  개인 개발자 레벨                                            │
│  TDD, 요구사항 문서화, 설계 문서, 테스트 문서                  │
│  → "개발자 한 명 한 명이 일하는 방식"                         │
├─────────────────────────────────────────────────────────────┤
│  ai-ops                                                     │
│  → 이 "개발자의 일하는 방식"을 LLM에게 이식                   │
└─────────────────────────────────────────────────────────────┘
```

**핵심 차이:**
- CI/CD: 팀/시스템 레벨에서 결과물 품질 관리
- ai-ops: 개인 개발자 레벨의 방법론을 LLM에게 이식

### 15.3 자동화 모델 (수정된 이해)

**이전 (잘못된 이해):**
- 사용자가 WP, ADR을 직접 작성
- ai-ops는 작성 여부만 검사

**현재 (올바른 이해):**
- 사용자는 업무 프로세스만 정의 (한 번)
- ai-ops가 WP, ADR 등 중간 기록을 **자동 생성**
- Gate는 자동 생성된 기록이 제대로 있는지 확인

```
┌─────────────────────────────────────────────────────────────┐
│  사용자 (1회)                                                │
│  "우리 팀은 이렇게 일해" → 프로세스 정의                       │
└─────────────────────────────┬───────────────────────────────┘
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ai-ops (매 작업마다 자동)                                    │
│  - 작업 시작 시 WP 자동 생성                                  │
│  - 의사결정 시 ADR 자동 기록                                  │
│  - 작업 완료 시 변경사항 자동 문서화                           │
│  - Gate로 누락 방지                                          │
└─────────────────────────────────────────────────────────────┘
```

### 15.4 타겟 사용자

- **1차 타겟**: 소규모 팀 (2~5명)
- **핵심 니즈**: 팀원이 어떤 바이브코딩 도구를 쓰든 같은 방법론으로 일하게 만들기

### 15.5 가치 제안 요약

| 관점 | 메시지 |
|------|--------|
| **기술적** | 개발 방법론을 LLM에게 이식하는 레이어 |
| **사용자** | 프로세스만 정의하면, 기록은 ai-ops가 알아서 한다 |
| **팀** | 누가 어떤 도구를 쓰든 같은 방법론으로 일하게 만든다 |

### 15.6 폐기된 비유들

| 비유 | 폐기 이유 |
|------|----------|
| "Process as Code" | IaC와의 유비가 부정확 (결정론적 vs 확률적) |
| "바이브코딩의 CI/CD" | CI/CD는 팀/시스템 레벨, ai-ops는 개인 레벨 |
| "바이브코딩의 SOP" | "강제/통제" 부정적 인식, 자동화 측면 미반영 |

## 16. MVP 스코프 확정 (2026-03-13)

### 16.1 핵심 방향 전환: 자체 프로세스 → 저명한 방법론 이식

**이전 (폐기):**
- 자체 정의 프로세스 P0-P4, S1-S4, G0
- 사용자가 새로운 프로세스 용어를 학습해야 함

**이후 (확정):**
- 이미 검증된 저명한 개발 방법론을 LLM에게 이식
- 사용자가 이미 아는 방법론 (TDD, Google Practices 등)
- "ai-ops는 방법론 정의자가 아니라 이식 엔진"

### 16.2 MVP 첫 번째 방법론: TDD

| 선택 이유 | 설명 |
|----------|------|
| **Gate 구현 명확** | Red → Green → Refactor가 기계적으로 검증 가능 |
| **결과물이 눈에 보임** | 테스트 코드 = 증거 |
| **성공 기준 객관적** | 테스트 통과/실패는 binary |
| **기존 인프라 재활용** | hook의 tool-level 차단으로 구현 가능 |

**Test-After (구현 먼저, 테스트 나중) 방식은 MVP에서 지원하지 않음:**
- "강제"할 게 없음 (구현 후 테스트 안 써도 막을 수 없음)
- TDD의 가치를 극적으로 보여주기 위해 집중

### 16.3 스코프 경계: 로컬 개발 세션

```
┌─────────────────────────────────────────────────────────────┐
│  ai-ops 영역 (로컬 개발 세션)                                 │
│                                                             │
│  ✅ 방법론 이식: TDD (Red-Green-Refactor)                    │
│  ✅ 문서 자동화: 요구사항 명세, 설계 문서                      │
│  ✅ 테스트 범위: Unit, Mock, SpringBootTest                  │
│  ✅ 검증 끝점: ./gradlew test (또는 npm test) 통과            │
├─────────────────────────────────────────────────────────────┤
│  CI/CD 영역 (ai-ops 범위 밖)                                 │
│                                                             │
│  ❌ 배포 파이프라인                                          │
│  ❌ E2E 테스트 오케스트레이션                                 │
│  ❌ 프로덕션 모니터링                                        │
└─────────────────────────────────────────────────────────────┘
```

**경계 설정 근거:**
- CI/CD는 팀/시스템 레벨, ai-ops는 개인 개발자 레벨
- GitHub Actions, Jenkins 등과 경쟁/중복 회피
- 명확한 정체성 유지

### 16.4 MVP 산출물

| 항목 | 설명 |
|------|------|
| **테스트 코드** | TDD 사이클의 직접적 결과물 |
| **요구사항 명세** | 작업 시작 시 자동 생성 |
| **설계 문서** | 의사결정 시 자동 기록 |
| **변경 이력** | 작업 완료 시 자동 문서화 |

### 16.5 MVP 성공 기준

1. **5분 내 첫 TDD 사이클 완료**: init → 첫 테스트 작성 → 구현 → 통과
2. **Gate 동작 체감**: "구현 먼저 쓰려니까 막힘 → 테스트 먼저 쓰니까 통과"
3. **문서 자동 생성 확인**: 아무것도 안 했는데 요구사항/설계 문서가 생성됨
4. **세션 재시작 후 컨텍스트 복구**: 어제 작업 맥락을 LLM이 기억

### 16.6 기술 과제 (MVP 전 해결 필요)

| 과제 | 우선순위 |
|------|----------|
| PyYAML 도입 (자체 파서 교체) | P0 |
| Bootstrap smoke test 추가 | P0 |
| TDD Gate 구현 (테스트 없이 구현 차단) | P0 |
| Hook latency 프로파일링 | P1 |
| Config schema 검증 연결 | P1 |

### 16.7 폐기된 대안

| 대안 | 폐기 이유 |
|------|----------|
| P0-P4 자체 프로세스 유지 | 학습 비용, "이게 뭔데?" 설득 필요 |
| Test-After 지원 | 강제할 게 없음, TDD 가치 희석 |
| 옵션 C (배포까지) | CI/CD 영역 침범, 정체성 흐려짐 |
| 멀티 방법론 동시 지원 | MVP 복잡도 증가, 희석 리스크 |

### 16.8 향후 확장 경로

1. **2차 방법론**: Google Engineering Practices (코드 리뷰)
2. **3차 방법론**: BDD (Behavior-Driven Development)
3. **방법론 플러그인 아키텍처**: 2차 방법론 추가 시점에 설계

## 17. Manus(Meta) 대비 포지셔닝 및 현실적 타겟 (2026-03-13)

### 17.1 Manus AI 개요

| 항목 | 내용 |
|------|------|
| **정의** | 세계 최초 완전 자율 AI 에이전트 |
| **인수** | Meta가 2025년 12월 인수 발표 |
| **기반** | Claude 3.5/3.7, Qwen 위 wrapper |
| **핵심 접근** | CodeAct (Python 코드로 액션 실행) |
| **핵심 가치** | "지시하면 알아서 해준다" |

### 17.2 ai-ops vs Manus: 스코프 차이

```
┌─────────────────────────────────────────────────────────────┐
│  Manus (작업 실행 레이어)                                     │
│  "이 보고서 작성해줘" → 자율 실행 → 결과물                     │
│  초점: "무엇을 할 것인가" (What)                              │
├─────────────────────────────────────────────────────────────┤
│  ai-ops (방법론 이식 레이어)                                  │
│  "TDD로 개발해" → TDD 사이클 강제 → 테스트 + 구현              │
│  초점: "어떻게 할 것인가" (How)                               │
└─────────────────────────────────────────────────────────────┘
```

| | Manus | ai-ops |
|---|---|---|
| **핵심 가치** | 작업 대신 수행 | 방법론 이식 + 강제 |
| **사용자 역할** | 지시 → 결과 확인 | 방법론 정의 → LLM이 따름 |
| **품질 보장** | 결과물 품질 | 과정 품질 |
| **타겟** | 일반 사용자 (비개발자 포함) | 소프트웨어 개발자/팀 |

### 17.3 대체 위험 평가

| 시나리오 | 위험도 | 분석 |
|----------|--------|------|
| Manus가 개발 방법론 강제 기능 추가 | ⚠️ 중간 | 가능하지만 Manus 핵심 가치와 충돌 |
| Manus가 "어떻게"는 무시 | ✅ 낮음 | ai-ops 영역 유지 |
| Manus + ai-ops 조합 사용 | 🤝 시너지 | Manus가 실행, ai-ops가 방법론 강제 |

**결론: 상호 보완적 관계**
- Manus: "뭘 할지"를 대신함
- ai-ops: "어떻게 할지"를 강제함
- Manus가 TDD로 해달라고 해도, 제대로 따르는지 검증/강제하는 것은 ai-ops 영역

### 17.4 현실적 타겟 재정의

**이전 (과대 목표):**
- 오픈소스 커뮤니티 확산
- 엔터프라이즈 도입
- 상용 소프트웨어화

**이후 (현실적 목표):**

| 타겟 | 설명 |
|------|------|
| **1차: 개인 재사용** | 새로운 프로젝트를 맡을 때마다 적용하는 개인 도구 |
| **2차: 사내 공유** | 팀 내 동료들에게 공유, N=2~5 검증 |
| **3차 (선택)**: 공개 | 검증 후 오픈소스 공개 고려 |

### 17.5 현실적 타겟의 장점

1. **부담 감소**: 외부 사용자 지원, 문서화, 커뮤니티 관리 부담 없음
2. **빠른 반복**: 내 필요에 맞게 자유롭게 수정 가능
3. **실용성 우선**: "남에게 보여주기 좋은 것"이 아니라 "내가 쓰기 좋은 것"
4. **검증 용이**: 직접 사용하면서 가치 검증

### 17.6 수정된 성공 기준

| 기준 | 측정 방법 |
|------|----------|
| **개인 재사용** | 새 프로젝트 3개 이상에서 ai-ops 적용 |
| **생산성 체감** | ai-ops 없이 작업할 때 불편함 느낌 |
| **사내 공유** | 동료 1명 이상이 자발적으로 채택 |
| **문서 최소화** | README + 5분 가이드면 시작 가능 |

### 17.7 Manus 시대에 ai-ops가 살아남는 이유

> **"Manus는 '해줄게'이고, ai-ops는 '이렇게 해'다."**

- Manus 사용자: "알아서 해줘" → 과정에 관심 없음
- ai-ops 사용자: "TDD로 해야 해" → 과정에 대한 conviction 있음

**ai-ops의 존재 이유**: 개발 방법론에 대한 신념이 있는 개발자/팀을 위한 도구. Manus가 아무리 발전해도 "방법론 강제"는 사용자가 원해야 하는 것이므로, ai-ops의 틈새는 유지됨.
