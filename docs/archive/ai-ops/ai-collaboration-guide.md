# Part 1. 운영 모델

---

## 1. 문서 목적

이 문서는 `ncube-regression-verify` 프로젝트를 AI 도구와 함께 수행하는 방식에 대해, 현재 합의된 지향점을 공식 기록물로 고정하기 위해 작성한다.

이 문서의 목적은 다음과 같다.

1. 현재 명확해진 프로젝트 운영 지향점을 유실 없이 보관한다.
2. Codex, Claude Code, 향후 추가될 AI 도구까지 포함하여 공통으로 따를 기준을 마련한다.
3. 즉흥적이고 세션 의존적인 바이브코딩 방식에서 벗어나, 프로세스 중심의 협업 체계로 전환하기 위한 상위 원칙을 선언한다.
4. 이후 작성될 프로세스 카탈로그, 작업 템플릿, 문서 구조, 도구별 모드/에이전트/스킬 정의 문서의 상위 기준 문서로 사용한다.

---

## 2. 배경

`ncube-regression-verify` 프로젝트는 일정 기간 동안 AI 도구를 적극적으로 활용하여 빠르게 기능을 구현하고 발전시켜 왔다.
이 방식은 초기 아이디어를 빠르게 실험하고 기능을 빠르게 형태화하는 데 매우 효과적이었다.

그러나 프로젝트 규모가 커지고, 작업 이력과 문서가 누적되면서 다음과 같은 한계가 명확해졌다.

- 세션이 바뀌면 컨텍스트가 잘린다.
- 변경 결과는 남아도 변경 의도와 판단 근거는 남지 않는 경우가 많다.
- AI가 브랜치, 런타임, 빌드 환경을 항상 안정적으로 이해하지 못한다.
- 기능 개발, 오류 수정, 문서화, 테스트 보강, 리팩토링, 리뷰 등의 후속 작업이 누락되기 쉽다.
- 문서가 많이 생성되었으나 역할 분리와 갱신 체계가 약해 중복과 불일치가 발생한다.
- 작업이 대화 중심으로 흘러가면서 프로젝트 차원의 기록성과 재현성이 약해진다.

따라서 앞으로는 AI를 단순한 보조 도구로 사용하는 수준을 넘어서,
**프로세스에 따라 움직이는 프로젝트 협업 주체**처럼 다루는 방향으로 운영 방식을 전환한다.

---

## 3. 최종 지향점

이 프로젝트의 최종 지향점은 다음과 같다.

> `ncube-regression-verify`에서 AI와 함께 수행하는 모든 개발 활동을
> "즉흥적 대화"가 아니라 "명시된 프로세스와 기록 가능한 산출물" 중심으로 운영한다.

즉, 앞으로의 작업은 다음 원칙을 따른다.

- 작업은 가급적 정의된 프로세스를 통해 시작한다.
- 작업은 단일 응답으로 끝나는 것이 아니라, 계획 → 실행 → 검토 → 기록의 흐름으로 다룬다.
- AI 도구는 단순 코드 생성기가 아니라 프로젝트 팀 내 역할을 수행하는 협업 주체처럼 활용한다.
- 세션이 바뀌어도 이어질 수 있도록 작업 맥락은 파일 기반 산출물로 남긴다.
- 작은 작업이라도 가능하면 리뷰와 기록을 거치며, 큰 작업은 중간 점검과 후속 점검을 포함한다.
- 문서화는 부가 작업이 아니라 프로젝트 산출물의 일부로 간주한다.

---

## 4. 핵심 철학

### 4.1 AI는 "즉석 답변 도구"가 아니라 "프로젝트 팀원"처럼 다룬다

AI 도구는 매번 새로 설명해야 하는 휘발성 도우미가 아니라,
프로젝트 운영 기준과 문서를 참조하여 일하는 가상 팀원처럼 활용한다.

이를 위해 다음을 지향한다.

- 프로젝트 차원의 공통 규칙을 먼저 정의한다.
- 특정 AI 도구에 종속되지 않는 공통 문서를 우선한다.
- 도구별 특성 차이는 별도 디렉토리 또는 별도 지침으로 분리한다.
- 사람과 AI 모두가 같은 문서를 보고 같은 기준으로 작업할 수 있게 한다.

---

### 4.2 작업은 프롬프트 단위가 아니라 프로세스 단위로 관리한다

앞으로의 작업은 "무엇을 할지 말하면 AI가 바로 구현하는 방식"보다,
"이 작업이 어떤 종류이며 어떤 절차를 밟아야 하는가"를 먼저 분류하고 시작하는 방식을 지향한다.

즉, 작업은 다음 질문에 답할 수 있어야 한다.

- 이 작업은 어떤 유형인가?
- 이 작업은 어떤 프로세스를 따라야 하는가?
- 선행되어야 하는 분석이나 구체화가 있는가?
- 병렬로 진행해야 할 보조 작업이 있는가?
- 완료 후 반드시 검토/기록/문서화해야 하는 항목은 무엇인가?

---

### 4.3 결과물뿐 아니라 의도와 판단 근거를 함께 남긴다

무엇을 바꿨는지만 남고 왜 그렇게 바꿨는지가 남지 않는 프로젝트는 시간이 지날수록 이해 비용이 커진다.
따라서 앞으로는 **변경 결과**와 함께 **변경 의도와 의사결정 논리**를 기록하는 것을 원칙으로 한다.

최소 기록 대상은 다음을 포함한다.

- 작업 배경
- 작업 목적
- 요구사항 또는 문제 정의
- 선택한 접근 방식
- 선택하지 않은 대안과 이유
- 완료 기준
- 검토 결과
- 후속 TODO 또는 남은 리스크

---

### 4.4 문서는 참고자료가 아니라 운영 자산이다

문서는 단순 부가 설명서가 아니라,
프로젝트를 이어갈 수 있게 만드는 **운영 자산이자 기록 자산**이다.

따라서 문서는 다음 역할을 수행해야 한다.

- 세션 전환 시 컨텍스트 유지
- AI 도구 전환 시 맥락 유지
- 의사결정 이력 보존
- 기능/구조/운영 방식 설명
- 외부 공유 자료의 원천 기록 역할 수행

---

### 4.5 작은 작업도 리뷰가 필요하다

본 프로젝트는 "한 번의 지시로 항상 원하는 결과가 나온다"는 가정을 두지 않는다.
따라서 작업 크기와 무관하게, 가능한 한 결과 검토 단계를 포함하는 것을 원칙으로 한다.

특히 다음 작업은 리뷰가 사실상 필수다.

- 핵심 로직 변경
- 설정 체계 변경
- 테스트 구조 변경
- 문서 계약 변경
- 외부 공개 문서 변경
- 리팩토링
- 회귀 가능성이 있는 수정

---

## 5. 프로젝트 운영 방식의 전환 목표

### 기존 방식
- 즉흥적인 대화 중심
- 세션 의존적
- 작업 맥락이 채팅에 흩어짐
- 후속 작업 누락 가능성 큼
- 문서가 많지만 체계는 약함
- 도구 전환 시 흐름이 끊김

### 목표 방식
- 프로세스 중심
- 파일 기반 기록 중심
- 세션 독립적
- 작업 유형별 표준 절차 존재
- 문서 구조와 역할이 분명함
- 도구가 바뀌어도 흐름이 이어짐

## 6. 프로세스, 모드, 에이전트의 개념 구분

이 프로젝트에서는 다음 세 개념을 구분한다.

### 6.1 프로세스
업무 절차 그 자체.
예: 기존 기능 분석, 요구사항 구체화, 신규 기능 개발, 오류 수정 개발

즉, **업무 흐름의 규약**이다.

### 6.2 모드
AI가 현재 어떤 업무 태도로 일할지를 정한 실행 상태.
예: Analysis Mode, Planning Mode, Troubleshooting Mode, Implementation Mode, Review Mode, Documentation Mode

즉, **행동 방식의 전환기**다.

### 6.3 에이전트/스킬
특정 프로세스를 빠르게 실행하기 위한 호출 단위.
예: `analyze-existing-feature`, `spec-refine`, `troubleshoot-root-cause`, `feature-delivery`, `bugfix-delivery`, `work-review`

즉, **실행 트리거**다.

### 6.4 관계 정리
- 프로세스는 상위 개념이다.
- 모드는 수행 태도다.
- 에이전트/스킬은 실행 인터페이스다.

---

## 7. 물리적 작업 환경 및 브랜치 운영 전략

AI 도구가 런타임 환경(Java 버전, Spring Boot 버전)을 혼동하거나 엉뚱한 설정 파일(`pom.xml` 등)을 수정하는 치명적인 오류를 원천 차단하기 위해, 본 프로젝트는 단일 디렉토리 내 브랜치 전환을 지양하고 **물리적으로 분리된 2개의 디렉토리**를 운영한다.

AI 도구는 현재 진입한 디렉토리의 성격에 따라 자신의 역할과 제약 사항을 명확히 인식해야 한다.

### 7.1 핵심 개발 환경 (`ncube-regression-verify-master`)

* **대상 브랜치:** `master`
* **실행 환경:** Java 17, Spring Boot 3.5.6
* **역할:** 프로젝트의 **Single Source of Truth (단일 진실 공급원)**이자 주 개발 환경

**운영 규칙:**

1. 모든 기능 개발(P3), 오류 수정(P4), 분석(P0), 설계(P1)는 **반드시 이 디렉토리에서 선행**되어야 한다.
2. 프로젝트의 **모든 공식 문서(작업 이력, 프로세스, 아키텍처 등)는 오직 master 브랜치에서만 관리**한다.
3. 이 환경에서 작업할 때 AI는 Java 17의 최신 문법과 Spring Boot 3.x의 최적화된 방식을 적극 활용한다.

### 7.2 레거시 백포팅 환경 (`ncube-regression-verify-java8`)

* **대상 브랜치:** `wafful4`
* **실행 환경:** Java 8, Spring Boot 2.3
* **역할:** `master` 브랜치에서 완료된 변경 사항을 레거시 환경으로 이관(Backporting)하기 위한 전용 환경

**운영 규칙:**

1. 이 디렉토리에서는 독창적인 신규 로직을 먼저 설계하거나 개발하지 않는다. 작업의 본질은 `master`의 코드를 Java 8 / Spring Boot 2.3 스펙에 맞게 **안전하게 다운그레이드 및 변환**하는 것이다.
2. 프로젝트의 공통 문서를 이 디렉토리에서 수정하거나 동기화하려 시도하지 않는다.
3. 이 디렉토리에는 오직 `wafful4` 브랜치 작업을 위한 AI 도구용 참고 정보(예: Java 8 전용 Maven 빌드 명령어, JDK 8 로컬 설치 경로, 레거시 호환성 주의사항 등)만 로컬 또는 도구 특화 설정(.codex/, .claude/ 등)으로 보관한다.

### 7.3 표준 작업 워크플로우 (Master-First Backporting)

기능 추가나 버그 수정이 발생할 경우, 작업은 다음 흐름을 엄격히 따른다.

1. `master` 디렉토리에서 프로세스(P0~P4)를 수행하고 변경과 문서화를 완전히 종료한다.
2. AI 도구의 작업 컨텍스트를 `wafful4` 디렉토리로 완전히 전환한다.
3. `master`의 변경분을 참고하여 `wafful4` 브랜치에 맞게 코드를 백포팅하고, 해당 환경 전용 명령어로 빌드 및 테스트(S1)를 수행하여 작업을 마무리한다.

# Part 2. 프로세스 카탈로그

---

## 8. 공통 운영 원칙

모든 프로세스는 다음 공통 원칙을 따른다.

### 8.1 프로세스 우선 원칙
작업은 가능한 한 "바로 구현"이 아니라 "적절한 프로세스 선택"부터 시작한다.

### 8.2 기록 우선 원칙
주요 작업은 채팅이 아니라 파일 기반 산출물로 남긴다.

### 8.3 리뷰 포함 원칙
작업 완료는 구현 종료가 아니라, 최소한의 검토와 기록이 끝난 상태를 의미한다.

### 8.4 문서 반영 원칙
코드 변경은 문서 영향 여부를 반드시 점검한다.

### 8.5 후속 프로세스 연결 원칙
각 프로세스는 가능한 후속 프로세스를 명시하고, 단독 수행으로 끝내지 않는다.

### 8.6 도구 독립 원칙
프로세스 정의는 특정 AI 도구에 종속되지 않는 공통 문서 기준으로 유지한다.

### 8.7 패킷 기반 상태 전이 원칙 (Packet-Based State Transition)

AI 도구의 세션 휘발성 및 도구 간(Codex ↔ Claude 등) 컨텍스트 단절을 극복하기 위해, 프로세스 간의 모든 데이터 전달은 채팅 프롬프트가 아닌 **'업무 패킷(Work Packet)'이라는 물리적 마크다운 파일**을 매개체로 사용한다.

**1. 영속성 강제 (Persistence First)**
어떤 프로세스(예: P0)든 작업을 완료하고 종료하기 위해서는, 반드시 해당 프로세스의 '필수 산출물'과 '후속 프로세스 지정' 내용을 업무 패킷 파일(예: `.ops/work-packets/WP-001-feature-x.md`)에 기록하고 저장해야만 완료된 것으로 간주한다.

**2. 파일 기반 컨텍스트 복원 (File-driven Context Restoration)**
후속 프로세스(예: P1)를 시작할 때, AI 도구는 사용자의 채팅 입력에 의존하는 대신 **직전 프로세스가 남겨둔 업무 패킷 파일을 최우선으로 읽어들여(Input)** 자신의 작업 상태와 컨텍스트를 스스로 복원해야 한다.

**3. 단일 진실 공급원 (Single Source of Truth for State)**
작업이 P0 → P1 → P3 → S1으로 흘러가는 동안, 현재 이 작업이 어느 단계에 머물러 있고 다음 단계는 무엇인지에 대한 상태(State) 정보는 오직 업무 패킷 파일 내의 메타데이터로만 관리한다.

### P0. 기존 기능 분석 프로세스

- 문서와 실제 구현의 정합성 확인
- 영향 범위가 넓은 변경

#### 10.3 시작 조건

- 현재 구조와 동작을 정확히 이해하지 못한 상태다.
- 구현 전에 현재 상태 파악이 필요한 것이 명확하다.

#### 10.4 입력

- 관련 이슈 또는 작업 배경

- 기존 테스트 정보

#### 10.5 수행 단계

7. 문서와 구현 간 불일치 여부를 확인한다.
8. 위험 포인트와 추가 확인 필요 항목을 정리한다.
9. 후속으로 적절한 프로세스를 제안한다.

#### 10.6 필수 산출물

- 현재 동작 요약

### P1. 요구사항 구체화 프로세스

#### 10.11 목적
추상적인 아이디어, 요청, 기대 결과를 **구현 가능한 요구사항**으로 정제하는 것을 목적으로 한다.

#### 10.12 이 프로세스를 사용하는 상황
- 하고 싶은 기능은 있지만 형태가 아직 모호함
- 구현 전에 범위와 완료 기준을 먼저 정리해야 함
- 의사결정 논리를 남기고 싶음
- 여러 구현 방안 중 하나를 선택해야 함

#### 10.13 시작 조건
- 해결하고 싶은 문제나 목표는 있으나 구현 단위로 명확하지 않다.
- 작업 전에 범위와 완료 기준을 정리할 필요가 있다.

#### 10.14 입력
- 아이디어 또는 요구
- 배경과 목적
- 관련 분석 결과(P0)
- 기존 제약사항
- 기대 결과
- 우선순위
- 알고 있는 리스크

#### 10.15 수행 단계
1. 작업 배경과 목적을 정리한다.
2. 해결하려는 문제를 명확히 정의한다.
3. 요구사항을 기능 단위로 분해한다.
4. 범위와 비범위를 구분한다.
5. 제약사항과 리스크를 정리한다.
6. 가능한 구현 접근 방안을 비교한다.
7. 완료 기준을 정의한다.
8. 미결 질문을 정리한다.
9. 후속 개발 프로세스로 넘길 입력값을 정리한다.

#### 10.16 필수 산출물
- 요구사항 정의
- 범위 / 비범위
- 제약사항
- 완료 기준
- 의사결정 근거
- 미해결 질문
- 후속 권장 프로세스

#### 10.17 병렬 가능 작업
- 관련 기존 문서 확인
- 관련 기능 분석(P0 결과 보강)
- 영향 테스트 현황 조사

#### 10.18 종료 조건
- 구현 대상이 명확하다.
- 완료 상태를 판별할 수 있다.
- 후속 개발에 필요한 입력이 충분하다.

#### 10.19 대표 후속 프로세스
- P3 신규 기능 개발
- P4 오류 수정 개발

### P2. 트러블슈팅 방법 도출 프로세스

#### 10.21 목적
문제를 바로 수정하기 전에, **원인 가설과 해결 전략**을 구조적으로 정리하는 것을 목적으로 한다.

#### 10.22 이 프로세스를 사용하는 상황
- 문제는 발견했지만 원인이 불명확함
- 재현은 되지만 수정 방향이 명확하지 않음
- 로그/테스트/설정/환경 등 복합 원인이 의심됨
- 곧장 수정하면 오진 위험이 큼

#### 10.23 시작 조건
- 증상이나 오류가 식별되었다.
- 원인 또는 수정 방향이 아직 확정되지 않았다.

#### 10.24 입력
- 증상
- 재현 정보
- 관련 로그 / 에러 / 관측 정보
- 관련 코드/문서/테스트
- 기존 기능 분석 결과(P0가 있다면 활용)

#### 10.25 수행 단계
1. 문제 증상을 명확히 기술한다.
2. 재현 조건과 비재현 조건을 정리한다.
3. 영향 범위를 추정한다.
4. 가능한 원인 가설을 목록화한다.
5. 각 가설의 확인 방법을 정의한다.
6. 해결안 후보를 정리한다.
7. 우선순위 해결 전략을 선택한다.
8. 수정 전에 필요한 추가 관측/로그/테스트를 정의한다.
9. 오류 수정 개발 프로세스로 넘길 입력값을 정리한다.

#### 10.26 필수 산출물
- 문제 정의
- 증상 / 재현 조건
- 영향 범위
- 원인 가설 목록
- 확인 방법
- 해결안 후보
- 우선 적용 전략
- 후속 오류 수정 입력값

#### 10.27 병렬 가능 작업
- 로그 보강 포인트 식별
- 재현 테스트 후보 정리
- 관련 문서 검토

#### 10.28 종료 조건
- 어떤 가설을 우선 검증할지 결정되었다.
- 바로 수정에 들어가도 될 정도로 방향이 정리되었다.
- 후속 오류 수정 프로세스 입력값이 준비되었다.

#### 10.29 대표 후속 프로세스
- P4 오류 수정 개발
- 추가 분석(P0 보강)
- 테스트 보강
- 관측/로그 보강

#### 10.30 체크리스트
- [ ] 증상이 명확히 기록되었는가
- [ ] 재현 조건이 정리되었는가
- [ ] 영향 범위를 정리했는가
- [ ] 원인 가설이 복수로 검토되었는가
- [ ] 확인 방법이 정의되었는가
- [ ] 해결 전략이 선택되었는가
- [ ] 후속 오류 수정 입력이 준비되었는가

### P4. 오류 수정 개발 프로세스

#### 10.41 목적
식별된 문제를 수정하고, 유사 문제의 재발을 방지하며, 수정의 타당성을 검증하는 것을 목적으로 한다.

#### 10.42 이 프로세스를 사용하는 상황
- 버그 수정
- 잘못 구현된 기능 보정
- 회귀 수정
- 예외/오동작/불일치 해결

#### 10.43 시작 조건

- `P1 -> P3 -> S1 -> S3 -> S2 -> S4`

### 12.2 기본 원칙

- 구현이 끝나면 S1, S2, S4 중 일부 또는 전부를 연결한다.
- 규모가 큰 작업이면 S3 리팩토링을 적극 고려한다.

## 14. 업무 패킷의 정의

업무 패킷은 프로젝트 내 하나의 작업을 설명하는 **최소 운영 단위**다.

업무 패킷은 단순 TODO가 아니라 다음을 포함하는 기록 단위다.

- 왜 시작되었는가
- 어떤 프로세스를 타고 있는가
- 지금 어디까지 진행되었는가
- 무엇이 산출되었는가
- 어떤 판단이 내려졌는가
- 다음에 무엇을 해야 하는가

즉, 업무 패킷은 **작업 설명 + 상태 + 산출물 + 이력 + 후속 연결 정보**를 묶은 단위다.

---

## 15. 업무 패킷 도입 원칙

### 15.1 모든 주요 작업은 업무 패킷으로 관리한다
작은 작업은 간단한 형태로, 큰 작업은 상세한 형태로 기록하되, 주요 작업은 가능한 한 업무 패킷으로 남긴다.

### 15.2 업무 패킷은 채팅 로그의 대체재가 아니라 상위 기록물이다
채팅은 과정의 일부일 뿐이며, 장기 보존과 재개 기준은 업무 패킷 문서를 따른다.

### 15.3 업무 패킷은 프로세스와 연결되어야 한다
각 업무 패킷은 시작 프로세스, 현재 프로세스, 후속 프로세스를 식별할 수 있어야 한다.

### 15.4 업무 패킷은 추적 가능해야 한다
관련 문서, 코드, 커밋, 테스트, 리뷰 결과와 연결될 수 있어야 한다.

### 15.5 업무 패킷은 상태 전이를 가진다
업무 패킷은 단순 "열림/닫힘"이 아니라, 분석/구체화/개발/리뷰/문서화/기록 등 실제 흐름을 반영하는 상태를 가진다.

## 22. 상태 의미 정의

### DRAFT
패킷이 생성되었지만 아직 시작 조건과 범위가 명확하지 않은 상태

### READY

### IN_REFINEMENT

### IN_TROUBLESHOOTING

### IN_DEVELOPMENT
P3 신규 기능 개발 또는 P4 오류 수정 개발 수행 중인 상태

### IN_REFACTORING
S3 리팩토링 수행 중인 상태

### IN_REVIEW
S1 작업 리뷰 수행 중인 상태

### IN_DOCUMENTATION
S2 문서 최신화 수행 중인 상태

### IN_RECORDING
S4 변경 이력 / 의사결정 기록 수행 중인 상태

### BLOCKED

### ON_HOLD

### DONE
정의된 완료 조건이 충족되고 후속 기록까지 끝난 상태

### CANCELLED
작업 자체가 무효화되었거나 더 이상 진행하지 않기로 결정된 상태

## 24. 상태 전이 세부 규칙

### T1. DRAFT -> READY
다음이 정리되면 가능하다.
- 작업 제목
- 목표
- 시작 프로세스
- 최소 범위
- 최소 완료 기준

### T2. READY -> IN_ANALYSIS
다음 중 하나면 진입한다.
- 기존 기능 이해가 필요함
- 영향 범위가 불명확함
- 레거시 구조 파악이 선행되어야 함

### T3. READY -> IN_REFINEMENT
다음 중 하나면 진입한다.
- 요구가 추상적임
- 완료 기준이 불명확함
- 범위/비범위 정리가 안 됨

### T4. READY -> IN_TROUBLESHOOTING
다음 중 하나면 진입한다.
- 문제 원인이 불명확함
- 오진 가능성이 큼
- 수정 전략이 정리되지 않음

### T5. READY -> IN_DEVELOPMENT
다음이 충족되면 가능하다.
- 구현 또는 수정 방향이 명확함
- 완료 기준이 존재함
- 최소 입력이 확보됨

### T6. IN_ANALYSIS -> IN_REFINEMENT
현재 상태를 파악했고, 이제 요구를 정교화해야 할 때 전이한다.

### T7. IN_ANALYSIS -> IN_TROUBLESHOOTING
현재 동작은 파악했으나 문제 원인/해결안 정리가 더 필요할 때 전이한다.

### T8. IN_ANALYSIS -> IN_DEVELOPMENT
분석만으로도 바로 구현 또는 수정 방향이 충분히 정리되었을 때 전이 가능하다.

### T9. IN_REFINEMENT -> IN_DEVELOPMENT
요구사항, 범위, 완료 기준이 준비되면 전이한다.

### T10. IN_TROUBLESHOOTING -> IN_DEVELOPMENT
우선 수정 전략이 선택되고 구현 가능한 입력값이 준비되면 전이한다.

### T11. IN_DEVELOPMENT -> IN_REVIEW
코드/설정/테스트 변경이 끝났고 검토 가능한 상태가 되면 전이한다.

### T12. IN_DEVELOPMENT -> IN_REFACTORING
구현은 되었지만 구조 정리가 필요하거나, 누적된 임시 구현이 커졌다면 전이한다.

### T13. IN_REVIEW -> IN_DEVELOPMENT
리뷰 결과 누락/오류/보완 필요 사항이 발견되면 재진입한다.

### T14. IN_REVIEW -> IN_DOCUMENTATION
리뷰가 통과했고 문서 반영 단계로 넘어갈 수 있으면 전이한다.

### T15. IN_DOCUMENTATION -> IN_RECORDING
문서 반영이 끝났고 최종 이력/의사결정 정리를 남길 단계면 전이한다.

### T16. IN_RECORDING -> DONE
다음이 충족되면 종료 가능하다.
- 필수 산출물 정리 완료
- 리뷰 결과 반영 완료
- 문서 반영 여부 기록 완료
- 변경/의사결정 기록 완료

### T17. 모든 진행 상태 -> BLOCKED
다음 중 하나면 BLOCKED로 전이 가능하다.
- 외부 정보 부족
- 결정권자 판단 대기
- 재현 실패
- 테스트 불안정
- 환경 이슈
- 범위 충돌

### T18. BLOCKED -> READY
막힘 원인이 해소되고 다시 시작 가능 상태가 되면 READY로 복귀한다.

### T19. 모든 상태 -> ON_HOLD
정책적 보류, 우선순위 조정, 릴리즈 일정 변경 등으로 잠시 중단할 때 전이한다.

### T20. READY/ON_HOLD/BLOCKED -> CANCELLED
작업 자체를 폐기하거나 더 이상 추진하지 않기로 결정되면 전이한다.

## 26. 업무 패킷 완료 기준

업무 패킷을 `DONE`으로 바꾸려면 최소한 다음 중 해당되는 조건이 충족되어야 한다.

### 공통
- 목표가 달성되었거나 종료 판단이 명확하다.
- 현재 상태와 결과를 이해할 수 있는 기록이 남아 있다.
- 후속 작업이 있다면 분리되어 있다.

### 구현 작업(P3/P4 포함 시)
- 코드 변경이 정리되었다.
- 리뷰가 수행되었거나 리뷰 생략 사유가 남아 있다.
- 문서 영향 여부가 기록되었다.
- 변경/의사결정 기록이 남아 있다.

### 분석/구체화 작업(P0/P1/P2 중심)
- 후속 프로세스로 넘길 입력값이 정리되었다.
- 분석/요구/트러블슈팅 결과가 기록되었다.
- 다음 단계 추천이 명시되었다.

- 작업 기간이 길어져 상태 추적이 어려워진다

## 31. 결론

이 문서는 `ncube-regression-verify` 프로젝트에서 AI 협업을 **프로세스 단위로 운영**하기 위한 통합 가이드다.

앞으로의 작업은 다음 원칙을 따른다.

- 작업은 가급적 정의된 프로세스를 통해 시작한다.
- 먼저 적절한 프로세스를 선택한다.
- 각 프로세스의 입력, 단계, 산출물, 종료 조건을 따른다.
- 구현은 전체 흐름의 일부로 본다.
- 리뷰, 문서화, 기록을 별도 후속 단계로 명시한다.
- 작업은 가능하면 업무 패킷으로 생성한다.
- 업무 패킷은 프로세스와 연결되어야 한다.
- 업무 패킷은 상태 전이를 통해 진행을 표현한다.
- 세션이 바뀌어도 이어질 수 있도록 작업을 기록 자산으로 남긴다.
- 완료는 "코드가 생김"이 아니라 "기록 가능한 종료 상태"를 의미한다.

이 문서는 향후 도구별 실행 규약과 작업 패킷 인덱스 체계의 기준 문서로 사용한다.
