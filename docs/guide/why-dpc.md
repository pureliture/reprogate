# Why dpc

## 문제

AI 기반 개발은 반복적으로 실패합니다:

- 세션 간 중요한 컨텍스트가 사라짐
- 메인 구현이 완료되어도 부수 작업(테스트, 문서)이 생략됨
- 의사결정 히스토리가 약해서 리뷰나 복구가 어려움
- 워크플로우가 특정 도구에 종속됨

## 해결책

dpc는 개발 방법론을 LLM에게 이식합니다.

세 가지 레이어로 동작합니다:

1. **guidelines** - 의도를 자연어로 전달 (LLM이 이해)
2. **rules** - 규칙을 강제 (Gate가 차단)
3. **presets** - 방법론을 패키지로 공유

## 왜 프롬프트만으로는 부족한가?

프롬프트 가이드만으로는 취약합니다.

좋은 프롬프트는 단일 세션을 개선할 수 있지만, 다음을 자동으로 제공하지 않습니다:

- 지속적인 기록
- 프로세스 규율
- 강제 가능한 체크포인트
- 도구 간 이식 가능한 행동

dpc는 일회성 프롬프팅 이상이 필요한 팀과 저장소를 위해 존재합니다.

## 누구를 위한 것인가

dpc는 다음을 중요시하는 사용자/팀에게 적합합니다:

- 작업 추적성
- 반복 가능한 실행
- 세션 간 연속성
- 도구 독립성
- 명시적 품질 게이트

가벼운 채팅 기반 워크플로우로 충분하다면 dpc는 과할 수 있습니다.

## dpc가 아닌 것

dpc는:

- 모든 프로젝트 관리 도구의 대체제가 아닙니다
- 모든 팀이 같은 프로세스 무게를 필요로 한다고 주장하지 않습니다
- 특정 벤더 종속 래퍼가 아닙니다
- 프로세스 강제만으로 엔지니어링 품질이 보장된다고 약속하지 않습니다

## 포지셔닝

dpc는:

- AI 기반 개발을 위한 방법론 이식 엔진
- 이식 가능한 프로세스 시스템
- 도구가 바뀌어도 개발 규율을 유지하는 방법
