# LLM 사용 시 프로세스 고정 오픈소스 안내?

## 한 줄 요약

**dpc는 `dev-ps-cast`의 줄임말로, 개발 방식 자체를 LLM에 붙여서 개인에게는 “내 방식대로 일하는 분신”을 만들고 팀에게는 “한 사람이 잡은 기준을 여러 사람이 비슷하게 따르게” 만드는 레이어다.**

여기서 이름은 이렇게 읽으면 됩니다.

- **dev**: development
- **ps**: process
- **cast**: 개발 프로세스를 LLM에 캐스팅한다는 의미

즉, **개발 프로세스를 LLM에 이식한다**는 쪽에 가까운 이름입니다.

---

## 뭐하는 친구인가?

dpc를 한 문장으로 설명하면,  
**“AI가 코드를 대신 써주는 것”을 넘어서, “어떻게 일해야 하는지”까지 같이 넘겨주는 방식**이라고 보면 됩니다.

이걸 개인 관점과 팀 관점으로 나눠 보면 더 이해가 쉽습니다.

### 개인이 쓸 때
개인에게 dpc는 **내 작업 방식을 LLM에 복제하는 도구**에 가깝습니다.

예를 들면:

- 나는 보통 테스트를 같이 챙긴다
- 큰 변경이면 문서도 같이 본다
- 중요한 판단은 이유를 남긴다
- 작업 흐름을 세션이 바뀌어도 이어가고 싶다

이런 습관을 매번 프롬프트로 다시 설명하지 않고,  
조금 더 구조적으로 붙여두고 싶을 때 dpc가 의미가 있습니다.

### 팀이 쓸 때
팀에서는 성격이 조금 달라집니다.

여기서는 **한 사람이 프로젝트 기준을 잘 셋업해두면, 나머지 사람들은 그 방법론을 깊게 학습하지 않아도 어느 정도 비슷한 결과물을 내게 만드는 것**에 가깝습니다.

즉:

- 리드나 숙련자가 기준을 정한다
- 그 기준을 dpc에 녹인다
- 다른 사람들은 그 기준을 따라 작업한다
- 결과물 편차를 줄인다

완전히 동일한 결과를 보장한다기보다는,  
최소한 **“누구는 테스트까지 하고 누구는 구현만 하고 끝나는 문제”** 같은 편차를 줄이는 쪽에 가깝습니다.

---

## dpc의 핵심 개념

dpc는 크게 세 가지로 설명할 수 있습니다.

### 1) guidelines
이건 **작업 원칙을 설명하는 층**입니다.

예를 들면:

- 테스트를 먼저 작성한다
- 구조 변경 시 문서를 같이 업데이트한다
- 중요한 변경은 배경을 남긴다

쉽게 말해 “우리는 보통 이렇게 일한다”를 적는 부분입니다.

### 2) rules
이건 **빠지면 안 되는 조건을 정의하는 층**입니다.

예를 들면:

- 특정 디렉토리의 코드를 바꾸면 테스트도 있어야 한다
- 공개 인터페이스를 바꾸면 문서 수정도 포함되어야 한다

guidelines가 작업 방식의 설명이라면,  
rules는 누락을 줄이기 위한 검사 조건입니다.

### 3) presets
이건 **guidelines와 rules의 조합을 재사용하기 위한 묶음**입니다.

예:

- TDD 중심 프리셋
- 문서/기록 중심 프리셋
- 최소 규칙 프리셋
- 팀 공통 개발 방식 프리셋

즉, dpc는 “좋은 프롬프트 모음”이라기보다  
**작업 방식을 설명 가능한 부분, 강제 가능한 부분, 재사용 가능한 부분으로 나눠서 다루는 구조**에 가깝습니다.

---

## 그럼 왜 이런 게 필요한가

AI 코딩 도구를 실제로 써보면, 속도와 별개로 반복해서 부딪히는 문제가 있습니다.

### 1) 세션이 바뀌면 작업 맥락이 잘 끊긴다
이전에는 분명 공유했던 배경이나 제약이  
다음 세션에서는 약해집니다.

그러면 AI가 지금 보이는 코드 기준으로는 그럴듯하게 작업해도,  
원래 의도와는 다른 방향으로 갈 수 있습니다.

### 2) 구현 외 작업이 자주 빠진다
기능은 만들어졌는데 이런 것들이 빠지기 쉽습니다.

- 테스트
- 문서 업데이트
- 변경 이유 기록
- 후속 검토 포인트 정리

사람은 보통 이걸 한 묶음으로 생각하지만,  
AI는 명시되지 않으면 구현만 하고 끝나는 경우가 많습니다.

### 3) 결과물 편차가 사람마다 커진다
이건 특히 팀에서 더 크게 드러납니다.

- 누군가는 테스트까지 챙기고
- 누군가는 구현만 하고
- 누군가는 문서를 같이 수정하고
- 누군가는 코드만 바꿉니다

AI를 쓰면 속도는 붙지만,  
오히려 **작업 방식의 편차**가 더 커질 수도 있습니다.

### 4) 결국 숙련자가 계속 직접 교정하게 된다
리드나 숙련자가 매번 리뷰에서 잡아주고,  
빠진 걸 메우고, 기준을 다시 설명해야 하면  
AI를 써도 운영비용이 줄지 않습니다.

dpc는 이 문제를  
**숙련자의 암묵지를 구조화해서 개인 또는 팀의 기본 작업 기준으로 내리는 방식**으로 풀어보려는 겁니다.

---

## dpc는 어떻게 동작하나

복잡하게 들릴 수 있지만 흐름은 단순합니다.

```text
작업 방식 정리
→ guidelines로 설명
→ rules로 필수 조건 정의
→ presets로 재사용 가능하게 묶음
→ AI 도구/훅/체크와 연결
→ 작업 수행
→ 누락이 있으면 감지하거나 막음
```

핵심은 두 가지입니다.

1. **권고와 강제를 분리한다**
   - 이해해야 하는 건 guidelines
   - 빠지면 안 되는 건 rules

2. **한 번 정리한 방식을 반복 가능하게 만든다**
   - 개인이면 내 작업 방식 복제
   - 팀이면 리드의 기준을 표준화

즉, dpc는 모델 성능을 바꾸는 게 아니라  
**AI가 일하는 운영 기준을 바꾸는 쪽**에 가깝습니다.

---

## 간단한 사용 예시

예를 들어 로그인 기능을 추가한다고 해보겠습니다.

### before
보통은 이렇게 요청합니다.

> 로그인 기능 추가해줘

그러면 기능 구현은 나올 수 있습니다.  
그런데 다음은 빠질 수 있습니다.

- 인증 관련 테스트
- 관련 문서 수정
- 왜 이 구조를 택했는지에 대한 기록

### after
dpc를 쓰면 요청 전에 작업 기준이 먼저 잡혀 있습니다.

예를 들면:

```yaml
preset: "tdd"

override:
  guidelines: |
    인증 관련 변경은 테스트와 문서 업데이트를 함께 수행한다.
    중요한 인증 흐름 변경은 이유를 기록한다.

rules:
  - trigger: "write"
    pattern: "src/auth/**/*.ts"
    require: "tests/auth/** exists"

  - trigger: "write"
    pattern: "src/auth/**/*.ts"
    require: "docs/** modified"
```

이 예시가 말하고 싶은 건 단순합니다.

- “이렇게 해주면 좋겠다”를 그냥 말로만 두지 않고
- “이건 빠지면 안 된다”까지 분리해서 다룬다

이렇게 해두면 개인은 매번 자기 방식을 설명하지 않아도 되고,  
팀은 리드가 잡은 기준을 다른 사람이 반복해서 따라가기가 쉬워집니다.

---

## CLAUDE.md, AGENTS.md와 뭐가 다른가

기존 방식이 쓸모 없다는 얘기는 아닙니다.  
오히려 CLAUDE.md나 AGENTS.md 같은 문서는 프로젝트 맥락을 전달하는 데 매우 유용합니다.

차이는 어디에 더 무게를 두느냐입니다.

### CLAUDE.md / AGENTS.md가 잘하는 것
- 프로젝트 설명
- 작업 규칙 안내
- 도메인 지식 전달
- 세션 시작 시 문맥 주입

### dpc가 추가로 하려는 것
- 지침과 규칙을 분리
- 재사용 가능한 preset 구조 제공
- 체크 가능한 조건 정의
- 개인 복제와 팀 표준화까지 염두에 둔 운영 모델 제공

간단히 말하면:

- **CLAUDE.md / AGENTS.md**는 설명 문서에 가깝고
- **dpc**는 설명 문서에 더해 **작업 규칙과 재사용 구조까지 같이 다루려는 접근**입니다

대체재라기보다는,  
기존 문서를 포함하면서 더 운영적으로 가져가려는 쪽에 가깝습니다.

---

## 누가 쓰면 좋나

dpc는 특히 아래 두 경우에 잘 맞습니다.

### 1) 개인이 자기 작업 방식을 LLM에 복제하고 싶을 때
매번 같은 기준을 프롬프트로 설명하는 게 번거롭고,  
세션이 바뀌어도 내 작업 방식을 유지하고 싶을 때 적합합니다.

### 2) 팀에서 한 사람이 기준을 잡고 결과물 편차를 줄이고 싶을 때
모든 팀원이 같은 수준으로 방법론을 익히길 기대하기 어렵다면,  
기준을 사람 머릿속이 아니라 작업 레이어 쪽으로 내리는 편이 현실적입니다.

조금 직설적으로 말하면,  
**리드 한 명이 계속 직접 캐리하는 대신, 그 캐리를 셋업으로 바꿔서 전파하고 싶은 경우**에 잘 맞습니다.

반대로,  
아주 짧은 실험 코드나 일회성 작업에는 다소 무겁게 느껴질 수 있습니다.

---

## 현재 상태

지금 dpc는 **문제 정의와 구조는 비교적 선명하지만, 제품 표면과 세부 구현은 계속 다듬는 단계**로 보는 게 맞습니다.

현재 비교적 정리된 것:

- 왜 필요한가에 대한 문제 정의
- guidelines / rules / presets라는 기본 모델
- 개인 복제 + 팀 표준화라는 사용 방향
- 어댑터 / 게이트 / 프리셋 중심의 구조 구상

진행 중인 것:

- presets 체계 구체화
- rules DSL 설계
- 도구별 어댑터 정리
- 실제 사용 흐름 다듬기

그래서 지금 시점에서는  
“이미 완성된 솔루션”이라고 말하기보다는,  
**이런 문제를 풀기 위한 프레임워크를 설계·구현 중인 상태**라고 보는 게 가장 정확합니다.

---

## 정리

dpc는 결국 두 가지를 같이 겨냥합니다.

- **개인에게는**  
  내 작업 방식을 복제해서 LLM을 좀 더 분신처럼 쓰게 하는 것

- **팀에게는**  
  한 사람이 셋업한 기준으로 다른 사람들도 러닝커브를 크게 들이지 않고 어느 정도 비슷한 결과를 내게 하는 것

그래서 dpc를 한 줄로 다시 정리하면 이 정도가 가장 맞는 것 같습니다.

> **dpc는 AI에게 코드를 맡기는 게 아니라, 작업 방식까지 같이 넘기기 위한 구조다.**
