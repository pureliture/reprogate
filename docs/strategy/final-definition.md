# ReproGate 최종 정의

## 핵심 전환

> **AI 협업을 대화 기반 감각 작업에서 재현 가능한 엔지니어링 체계로**

## Core Identity

> **ReproGate는 작업 기록을 필수 산출물로 삼아, AI와 협업하며 얻은 작업 패턴을 Skill로 누적하고 이를 규칙으로 강제하는 개인·팀용 방법론 컴파일러다.**

ReproGate의 핵심은 단순한 프롬프트 모음도, 무거운 상태 추적 오케스트레이터도 아니다. ReproGate는 **작업 패턴을 기록 가능한 형태로 자산화하고**, 그 기록 위에서 **규칙을 평가하고 강제하는 Compiler + Gatekeeper**다.

## 왜 기록이 중심인가

작업 패턴을 규칙으로 강제하려면, 규칙이 검사할 수 있는 **물리적 증거**가 필요하다. 기억, 대화 맥락, 세션 내부 상태만으로는 강제할 수 없다.

예를 들어:

- 설계를 먼저 하라는 패턴은 설계 산출물이 있어야 검사할 수 있다.
- 중요한 판단 이유를 남기라는 패턴은 의사결정 기록이 있어야 검사할 수 있다.
- 작업 목표와 완료조건을 분명히 하라는 패턴은 작업 계획과 진행 기록이 있어야 검사할 수 있다.
- 변경 이유와 결과를 남기라는 패턴은 변경 및 검증 흔적이 있어야 검사할 수 있다.

즉, **기억 기반 작업은 강제할 수 없고, 기록 기반 작업만 강제할 수 있다.**

그래서 ReproGate에서 **작업 기록 저장은 부가 기능이 아니라 기본이자 필수 기능**이다.

## ReproGate가 말하는 “작업 기록”

ReproGate에서 말하는 작업 기록은 단순한 세션 상태 저장이 아니다. 그것은 **의도, 범위, 결정, 완료조건, 변경 근거를 남기는 공학적 기록**이다.

대표적인 기록 범주는 다음과 같다:

- **작업 계획 기록**
  - 무엇을 하려는가
  - 범위는 어디까지인가
  - 완료 기준은 무엇인가
- **의사결정 기록**
  - 왜 이런 판단을 했는가
  - 어떤 대안을 검토했고 왜 버렸는가
- **설계·변경·검증 기록**
  - 무엇이 바뀌었는가
  - 무엇이 증명되었는가

이 기록들은 단순 보관용 메모가 아니라, **AI 작업을 설명 가능하고 검증 가능하게 만드는 기준선**이다.

## 상태 저장과의 차이

세션 정보나 현재 상태를 저장하는 기능만으로는 다음을 알기 어렵다:

- 바이브 코딩이 어디서부터 잘못 흘러갔는지
- 설계가 어떤 판단 지점에서 꼬였는지
- LLM이 왜 그런 선택을 했는지
- 원래 의도와 결과가 왜 어긋났는지

그래서 ReproGate는 단순 상태 저장과 별도로, **작업 기록**을 명시적으로 요구한다.

정리하면:

- **상태 저장**
  - 현재 상태, 세션 재개, 런타임 보조
  - “지금 어디까지 왔는가”
- **작업 기록**
  - 의도, 판단, 근거, 완료조건
  - “왜 이렇게 일했고 어디서 어긋났는가”

> **상태 저장은 재개를 돕고, 작업 기록은 판단을 설명한다. ReproGate의 핵심은 후자다.**

## 지식의 저장 구조: Skill의 누적

ReproGate는 작업 과정에서 발견된 좋은 패턴을 **Skill**로 축적한다.

### Skill

하나의 완결된 작업 패턴 단위다.

구성:

- **guidelines.md** — 의도, 원칙, 기대 행동
- **rules.rego** — 물리적 강제 규칙

핵심은, 작업하면서 얻은 노하우가 대화 속에서 휘발되지 않고 **텍스트 자산으로 누적된다는 것**이다.

## Pattern / Preset: Skill의 조합

특정 작업 유형은 여러 Skill의 조합으로 정의된다.

예:

- 신규 기능 개발
- 버그 수정
- 리팩토링
- 설계 변경

각 작업은 별도의 거대한 상태 머신으로 추적되는 것이 아니라, **이 작업에는 어떤 Skill들이 필수인가**의 조합으로 구성된다.

즉 ReproGate는 복잡한 중앙 오케스트레이터보다, **평탄화된 Skill 집합과 그 조합**에 더 가깝다.

## 핵심 메커니즘: 기록 → Skill → Rule → 강제

ReproGate의 흐름은 다음과 같다.

1. 작업이 진행된다.
2. 작업 중 의도, 결정, 범위, 근거가 기록된다.
3. 반복되는 좋은 패턴이나 자주 놓치는 누락이 발견된다.
4. 그 패턴이 Skill로 승격된다.
5. Skill의 규칙이 이후 작업에서 Gate로 강제된다.

즉:

- **기록은 증거**
- **Skill은 자산화된 패턴**
- **Rule은 강제 메커니즘**
- **Gate는 물리적 통제 장치**

이 구조를 통해 ReproGate는 AI 협업을 대화 기반 감각 작업에서 **재현 가능한 엔지니어링 체계**로 바꾼다.

## Artifact-Driven Workflow

ReproGate는 “지금 몇 단계인가” 같은 무거운 상태 추적 자체를 제품 정체성의 중심에 두지 않는다. 대신 **현재 존재하는 산출물과 규칙의 조합**이 자연스럽게 작업 순서를 만든다.

예:

- 구현하려고 함 → 설계 산출물 없음 → 차단
- 코드 수정함 → 검증 흔적 없음 → 차단
- 마무리하려고 함 → 변경 기록 없음 → 차단

결과적으로 ReproGate 코어는 단순히 규칙만 평가하지만, 밖에서 보면 **작업 순서가 강제되는 것처럼 보이는 오케스트레이션 효과**가 나타난다.

즉 ReproGate는 **state-driven orchestrator**라기보다 **artifact-driven compiler/gatekeeper**다.

## 팀 레벨의 의미

ReproGate는 개인용 가드레일에 머물지 않는다.

> **ReproGate는 개인이나 팀이 AI와 협업하며 얻은 작업 패턴을 Skill로 자산화하고, 이를 저장소에 동기화된 규칙으로 강제하여 결과물 편차를 줄이는 팀 운영 표준이다.**

팀 리드나 숙련자가 AI와 협업하며 다듬은 패턴은 잔소리나 휘발성 대화로 남지 않고, 저장소의 텍스트 자산으로 축적된다.

그리고 팀원은 이를 깊이 암기하지 않아도 된다. 도구는 그 기록과 규칙을 읽고, **리드가 설계한 작업 방식으로 수렴**하게 된다.

결과적으로 ReproGate는 사람의 의지 대신 시스템 규칙으로 작업 품질과 결과물 편차를 평준화한다.

## ReproGate는 무엇이 아닌가

ReproGate는:

- 무거운 상태 추적 오케스트레이터가 아니다
- 대화 기억만 잘 보존하는 메모 시스템이 아니다
- 문서를 많이 쓰게 만드는 관리 도구가 아니다
- 특정 벤더 전용 래퍼가 아니다
- 프롬프트만 모아놓은 템플릿 저장소가 아니다

## ReproGate가 집중하는 것

ReproGate가 집중하는 것은 다섯 가지다:

1. **기록**
   - 작업의 의도와 범위
   - 진행 상태와 완료 조건
   - 의사결정 근거
   - 설계와 검증 흔적
2. **Skill 축적**
   - 작업 패턴의 자산화
3. **Guarantee**
   - Rego 기반 규칙 강제
4. **Artifact-Driven Workflow**
   - 산출물 기반 흐름 유도
5. **팀 표준화**
   - 개인 패턴을 팀 운영 표준으로 확장

## 최종 정의

> **ReproGate는 작업 기록을 필수 산출물로 남기게 하고, 그 기록에서 얻은 작업 패턴을 Skill로 누적하며, 이를 규칙으로 강제함으로써, 개념과 기억에 의존하는 AI 코딩을 기록으로 증명하는 AI 엔지니어링으로 전환하는 방법론 컴파일러다.**
