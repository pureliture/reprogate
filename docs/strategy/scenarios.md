# LLM Quick Guide

> 이 섹션은 긴 시나리오 카탈로그를 LLM이 빠르게 해석할 수 있도록 돕는 압축 헤더다.
> 본문 전체를 대체하지 않으며, 해석 우선순위와 핵심 불변 원칙만 요약한다.

---

## 1. ReproGate 한 줄 정의

**ReproGate는 작업 기록을 필수 산출물로 삼고, AI 협업에서 나온 판단·패턴을 Skill로 축적한 뒤 Rule과 Gate로 연결해 재현 가능한 작업 방식을 구조화·강제하는 delivery harness다.**

---

## 2. 이 문서의 목적

이 문서는 ReproGate의 제품 바운더리를 **실제 사용자 시나리오** 기준으로 정의한다.

LLM은 이 문서를 읽을 때 다음 질문에 답해야 한다.

1. 이 상황에서 ReproGate가 **반드시 책임져야 하는 것**은 무엇인가
2. 사용자가 우회하거나 자유롭게 바꿔도 되는 것은 무엇인가
3. 어떤 상태가 **절대 허용 불가한 계약 위반**인가
4. 이 시나리오가 어떤 **핵심 계약 태그**(`RC`, `DT`, `VT`, `ST`, `WD`, `GA`, `PI`, `PS`, `PV`)에 연결되는가

---

## 3. 절대 불변 해석 원칙

LLM은 아래 원칙을 문서 전체 해석의 최우선 기준으로 삼아야 한다.

### P-1. Built-in 사용 여부가 아니라 재현성 결손을 본다
- Workflow를 안 썼다고 바로 실패가 아니다
- Built-in Skill을 안 썼다고 바로 실패가 아니다
- 대신 Record, Decision, Verification, Gap, Policy Input 같은 **재현성 계약 누락**을 본다

### P-2. Deviation은 허용하되 Silent Loss는 금지한다
- 단계 생략, 계획 변경, 수동 수정, 외부 작업은 허용된다
- 하지만 그 과정이 기록 없이 사라지면 안 된다

### P-3. Freeform은 허용하되 무구조 자동 통과는 금지한다
- 자유 대화와 자유 작업은 가능하다
- 다만 최종적으로는 필요한 수준의 구조화, 또는 명시적 예외 승인 경로가 필요하다

### P-4. Late Binding은 허용하되 Provenance 없는 사후 서사는 금지한다
- 나중에 Record를 정리하거나 보완할 수 있다
- 그러나 소급 작성된 결론은 반드시 원본 Trace / Commit / Diff / Artifact와 연결돼야 한다

### P-5. Built-in은 기본값이지 유일한 표준이 아니다
- Workflow, Skill, Storage, Rule은 커스터마이징될 수 있다
- 단, 어떤 경우에도 core contract는 유지되어야 한다

---

## 4. 문서 읽는 순서

LLM은 질문 유형에 따라 아래 순서로 읽는다.

### 작업 수행 흐름을 묻는 경우
- **1부. 작업 수행 시나리오** 우선 참조
- 특히 `SC-1.x`, `SC-2.x`, `SC-3.x`, `SC-4.x`

### 구조화된 문서 계약 / Gate / Rule을 묻는 경우
- **2부. 구조화된 기록과 정책 평가 시나리오** 우선 참조
- 특히 `SC-5.x`, `SC-6.x`

### 저장 방식 / 커스터마이징 / 외부 시스템 연동을 묻는 경우
- **3부. 저장 방식 및 커스터마이징 시나리오** 우선 참조
- 특히 `SC-7.x`

### Skill / Workflow 추출, 작성, 수정, 파생을 묻는 경우
- **4부. 방법론 추출·작성·수정 시나리오** 우선 참조
- 특히 `SC-8.x`, `SC-9.x`

### Built-in vs Custom 충돌, 팀 규칙, 승인 흐름을 묻는 경우
- **5부. Built-in / Custom / Governance 시나리오** 우선 참조
- 특히 `SC-10.x`, `SC-11.x`

### 다른 프로젝트 이식 / Pack 재사용 / 공통 계약을 묻는 경우
- **6부. 포팅 / 이식 / 프로젝트 확장 시나리오** 우선 참조
- 특히 `SC-12.x`

### 구현 리스크 / 잊으면 안 되는 설계 보강점을 묻는 경우
- **부록 B. 시나리오에서 파생된 설계 보강 포인트** 우선 참조
- 특히 `D-01` ~ `D-06`

## 5. 가장 중요한 시나리오 우선순위

LLM이 전체 문서를 다 다루기 어려울 때, 아래 시나리오를 최우선 핵심 시나리오로 간주한다.

- `SC-1.3` : Workflow 단계 생략과 Deviation
- `SC-2.4` : Skill 결과 수동 Override
- `SC-3.2` : Freeform 종료 후 Late Binding
- `SC-4.2` : 외부 작업의 사후 등록
- `SC-5.1` : 자유 Markdown vs 구조화 계약
- `SC-6.3` : Hard Gate 직전 일괄 보완
- `SC-7.3` : Built-in 저장 거부 + Gate 사용
- `SC-8.2` : 유사 Skill 충돌
- `SC-9.3` : Built-in Workflow 파생
- `SC-10.1` : Built-in vs Custom Skill 충돌
- `SC-11.1` : 개인 Skill의 팀 승격
- `SC-12.2` : 저장 방식이 달라도 Rule 재사용

---

## 6. 핵심 계약 태그 해석

LLM은 시나리오를 요약할 때 가능하면 아래 태그 기준으로 정리한다.

- `RC` = 작업 기록 단위가 존재하고 식별 가능해야 함
- `DT` = 의사결정 근거가 남아야 함
- `VT` = 검증 흔적과 증빙이 남아야 함
- `ST` = Skill 실행/적용 이력이 남아야 함
- `WD` = Workflow 편차가 기록돼야 함
- `GA` = Gap 또는 예외가 명시적으로 승인돼야 함
- `PI` = 정책 평가 가능한 구조가 준비돼야 함
- `PS` = scope / precedence / governance가 명확해야 함
- `PV` = 다른 프로젝트로의 이식 가능성이 유지돼야 함

---

## 7. 답변 시 금지할 오해

LLM은 아래와 같은 오해를 하면 안 된다.

- “Built-in workflow를 안 썼으니 실패”
- “자유 대화만 했으니 무조건 통과 불가”
- “Markdown 본문만 있으면 구조 계약도 충족한 것”
- “사후 문서화는 무엇이든 나중에 그럴듯하게 적으면 된다”
- “Skill이 한 번 실행됐으니 결과는 자동으로 신뢰 가능”
- “개인 Skill은 바로 팀 표준이 될 수 있다”
- “저장 방식이 다르면 Rule/Gate를 공통으로 못 쓴다”

---

## 8. 답변 형식 권장

LLM은 이 문서를 근거로 답변할 때 아래 형식을 우선 사용한다.

1. **해당 질문이 속한 시나리오/부**
2. **ReproGate 필수 책임**
3. **허용 가능한 우회**
4. **절대 허용 불가한 계약 위반**
5. **관련 계약 태그**
6. **필요 시 부록 B의 설계 보강 포인트 연결**

---

## 9. 최종 해석 기준

이 문서 전체는 다음 한 문장으로 수렴한다.

> **ReproGate는 사용자가 workflow, skill, freeform, external storage 중 어떤 방식으로 시작하든, 그 작업을 추적 가능한 기록 단위로 붙잡고 필요 시 구조화하여 Skill과 Rule로 연결함으로써 재현 가능한 작업 방식을 구조화·강제해야 한다.**

# 문서 해석 원칙

## 1. ReproGate는 도구 사용 여부가 아니라 재현성 결손을 평가한다
- Workflow를 안 썼다고 바로 차단하지 않는다.
- Built-in Skill을 안 썼다고 바로 실패시키지 않는다.
- 대신 Record, Decision, Verification, Gap Acknowledge, Policy Input 상태 같은 **재현성 계약**이 충족되는지를 본다.

## 2. Deviation은 허용하되 Silent Loss는 허용하지 않는다
- 단계 생략, 순서 변경, 수동 덮어쓰기, 외부 작업, 구조 해제는 가능하다.
- 그러나 그 과정이 기록 없이 사라지거나, 이유 없이 최종 Gate를 통과하는 것은 허용하지 않는다.

## 3. Freeform은 허용하되 무구조를 기본 통과 기준으로 삼지 않는다
- 자유 대화, 자유 작업은 허용된다.
- 다만 최종적으로는 필요한 수준의 구조화된 Record로 승격되거나, 부족분에 대한 명시적 소명·예외 승인 경로가 있어야 한다.

## 4. Built-in은 기본 제공값이지 유일한 정답이 아니다
- Workflow, Skill, Storage, Rule, Visualizer는 커스터마이징 가능하다.
- 다만 어떤 방식이든 core contract는 유지되어야 한다.

## 5. Late Binding은 허용되지만 Provenance 없는 사후 서사는 허용하지 않는다
- 나중에 Record를 정리하거나 일괄 보완할 수 있다.
- 그러나 소급 작성된 결론은 반드시 원본 Trace, Commit, Diff, 로그 등과 연결되어야 한다.

# 공통 계약 태그

시나리오별 판정 근거로 아래 태그를 사용한다.

- `RC` = Record Contract 존재
- `DT` = Decision Trace 존재
- `VT` = Verification Trace 존재
- `ST` = Skill Trace 존재
- `WD` = Workflow Deviation 기록
- `GA` = Gap Acknowledgement 존재
- `PI` = Policy Input Ready (정책 평가 가능한 구조화 상태)
- `PS` = Precedence / Scope 규칙 적용
- `PV` = Portability / Project Variance 대응 가능

# 1부. 작업 수행 시나리오

사용자가 ReproGate를 통해 실제 작업을 수행할 때 진입하고 진행하는 방식에 대한 시나리오들.

---

## 1. Workflow-first

사전에 정의된 단계적 절차(Workflow)를 기반으로 작업을 시작하는 경우.

### SC-1.1: 끝까지 All Accept로 플로우대로 진행
#### 맥락
- **시작 모드**: workflow
- **전환 여부**: 없음
#### 사용자의 거부 포인트
- 없음
#### ReproGate 필수 책임
- 선언된 워크플로 단계별 순차 진행 안내 및 상태 추적
- 각 단계 산출물이 표준 Record 구조를 충족하는지 검증
- 종료 시 Gate 평가 수행
#### 허용 가능한 우회
- 없음
#### 절대 허용 불가한 계약 위반
- 최종 단계 종료 전에 필수 Context/Decision/Verification 누락
#### 관련 계약 / Rule
- `RC`, `DT`, `VT`, `PI`

---

### SC-1.2: 중간에 플랜 변경이 N번 발생
#### 맥락
- **시작 모드**: workflow
- **전환 여부**: 없음 (방향 전환만 발생)
#### 사용자의 거부 포인트
- “아까 내린 아키텍처 결정은 틀렸어. 다시 설계 방향을 수정할래.”
#### ReproGate 필수 책임
- 이전 기록을 덮어쓰지 않고 변경 이력으로 누적
- 변경된 계획이 후속 검증 대상에 반영되도록 Rule 재평가
- Change Rationale을 명시적으로 남기게 함
#### 허용 가능한 우회
- 이전 단계로 되돌아가 재작성
- 기존 초안을 폐기하고 새 초안으로 대체
#### 절대 허용 불가한 계약 위반
- 플랜 변경 사실은 있으나, 왜 변경했는지에 대한 근거가 없음
#### 관련 계약 / Rule
- `RC`, `DT`, `WD`, `PI`

---

### SC-1.3: 중간 단계 Bypass
#### 맥락
- **시작 모드**: workflow
- **전환 여부**: workflow 내 Deviation
#### 사용자의 거부 포인트
- “이 단계는 우리 프로젝트에 불필요하니 생략할래.”
#### ReproGate 필수 책임
- 단계 생략을 에러로 간주하지 않고 Deviation으로 기록
- 생략 후에도 재현성 계약이 유지되는지 재평가
- 생략된 단계 목적을 대체하는 다른 증빙이 있는지 확인
#### 허용 가능한 우회
- 특정 산출물 형식 대체
- 다단계 압축
#### 절대 허용 불가한 계약 위반
- 핵심 검증 단계를 생략했는데 대체 검증 근거가 전혀 없음
#### 관련 계약 / Rule
- `WD`, `VT`, `PI`

---

### SC-1.4: 중도 종료 후 Trace-only 전환
#### 맥락
- **시작 모드**: workflow
- **전환 여부**: workflow → trace-only
#### 사용자의 거부 포인트
- “워크플로 너무 복잡해. 구조는 끄고 그냥 대화하면서 진행할래.”
#### ReproGate 필수 책임
- Workflow 상태 추적은 중단하되 Raw Trace 수집은 지속
- 현재부터는 정식 단계 추적이 아니라 trace-only 상태임을 명시
- 이후 Late Binding 또는 결손 소명 경로 제공
#### 허용 가능한 우회
- 구조화 문서 생성을 나중으로 미룸
#### 절대 허용 불가한 계약 위반
- 중도 이탈 후 아무런 승격/소명 없이 최종 Gate 통과
#### 관련 계약 / Rule
- `RC`, `GA`, `PI`

---

### SC-1.5: 중도 종료 후 완전 Bypass 전환
#### 맥락
- **시작 모드**: workflow
- **전환 여부**: workflow → bypass
#### 사용자의 거부 포인트
- “이후 작업은 ReproGate 없이 진행할래.”
#### ReproGate 필수 책임
- 구조 중단과 추적 중단을 구분된 상태로 다룸
- 중단 시점까지의 상태를 고정하고 Gap 가능성을 표시
- 복귀 시 사후 복기 또는 Gap Acknowledge 경로 제공
#### 허용 가능한 우회
- 중단 이후 외부 도구 사용
#### 절대 허용 불가한 계약 위반
- 중단 구간이 있었는데 아무런 Acknowledge 없이 최종 Gate 통과
#### 관련 계약 / Rule
- `GA`, `WD`

---

## 2. Skill-first

특정 목적을 가진 단일 패턴/기능(Skill) 호출을 중심으로 작업하는 경우.

### SC-2.1: 단발성 Skill 호출 후 종료
#### 맥락
- **시작 모드**: skill
- **세션 길이**: 단발
#### 사용자의 거부 포인트
- “전체 구조는 필요 없고 `check-lint`만 한 번 돌리고 끝낼래.”
#### ReproGate 필수 책임
- Workflow가 없어도 Skill 실행 이력과 결과를 단독 Trace로 기록
- 어떤 파일/맥락/작업 단위와 연결되는지 최소 연결 유지
#### 허용 가능한 우회
- 별도 Record 문서를 즉시 생성하지 않고 세션 Trace에 연결
#### 절대 허용 불가한 계약 위반
- Workflow가 없다는 이유로 Skill 호출 이력이 사라짐
#### 관련 계약 / Rule
- `ST`, `RC`

---

### SC-2.2: 멀티 Skill과 자유 대화의 교차
#### 맥락
- **시작 모드**: skill + trace-only 혼합
- **세션 길이**: 장기
#### 사용자의 거부 포인트
- “필요할 때 스킬 쓰고, 막히면 30분 대화하고, 또 다른 스킬 쓸래.”
#### ReproGate 필수 책임
- 파편화된 Skill 호출과 자유 대화를 하나의 연속 타임라인으로 관리
- Decision / Verification이 Skill 구간과 단절되지 않게 연결
- 현재 맥락에서 추천되는 다음 Skill 또는 누락 상태 표시
#### 허용 가능한 우회
- 스킬 간 긴 공백 대화
#### 절대 허용 불가한 계약 위반
- 자유 대화 구간에서 발생한 중요한 결정이 Skill 기록과 단절됨
#### 관련 계약 / Rule
- `ST`, `DT`, `RC`, `PI`

---

### SC-2.3: Skill 반복 중 Workflow로 승격
#### 맥락
- **시작 모드**: skill → workflow
#### 사용자의 거부 포인트
- “처음엔 작은 수정인 줄 알았는데 규모가 커졌어. 정식 워크플로로 바꿔줘.”
#### ReproGate 필수 책임
- 기존 Skill 이력과 대화 맥락을 유실하지 않음
- 이전 Trace를 새 Workflow의 초기 Context / Decision 초안으로 병합
- 승격 전후의 기준 차이를 명시
#### 허용 가능한 우회
- 초기 단계 일부 자동 충족 처리
#### 절대 허용 불가한 계약 위반
- 승격 과정에서 선행 Skill 이력과 판단 근거가 사라짐
#### 관련 계약 / Rule
- `ST`, `RC`, `DT`, `PI`

---

### SC-2.4: Skill 결과를 수동 수정으로 덮어쓰기
#### 맥락
- **시작 모드**: skill
- **전환 여부**: 없음
#### 사용자의 거부 포인트
- “스킬 결과가 마음에 안 드니 그냥 내가 수동으로 다시 고칠게.”
#### ReproGate 필수 책임
- Skill 결과의 폐기/부분 채택 여부를 기록
- 최종 결과가 Skill 결과와 어떻게 달라졌는지 연결
- 필요 시 Manual Override 의심 상태를 감지하고 사용자 확인 유도
#### 허용 가능한 우회
- Skill 산출물 일부만 채택
#### 절대 허용 불가한 계약 위반
- Skill 실행 이력은 있는데 최종 결과와의 관계가 설명되지 않음
#### 관련 계약 / Rule
- `ST`, `DT`

---

### SC-2.5: Skill 결과를 받아들이지 않았지만 이유만 남기고 재실행하지 않음
#### 맥락
- **시작 모드**: skill
#### 사용자의 거부 포인트
- “이 결과는 안 맞아. 이유만 적고 내 방식으로 갈래.”
#### ReproGate 필수 책임
- Skill reject 이유를 Decision Trace로 남김
- 이후 수동 경로가 별도 검증 필요 대상임을 표시
#### 허용 가능한 우회
- 재실행 없이 수동 수정
#### 절대 허용 불가한 계약 위반
- Skill reject 후 후속 검증 없이 동일 수준 신뢰로 취급
#### 관련 계약 / Rule
- `ST`, `DT`, `VT`

---

## 3. Trace-only (Freeform)

어떤 템플릿이나 스킬, 워크플로의 간섭 없이 추적 기능만 활성화한 채 자유롭게 작업하는 경우.

### SC-3.1: 구조 없이 자유 대화 누적
#### 맥락
- **시작 모드**: trace-only
#### 사용자의 거부 포인트
- “아무 템플릿도 빈 파일도 만들지 말고 그냥 대화만 해.”
#### ReproGate 필수 책임
- Workflow/Skill 권유를 강제 팝업하지 않음
- 시스템 백그라운드에서 Raw Trace만 수집
- 추후 구조화 가능한 최소 연결 정보 보존
#### 허용 가능한 우회
- 공식 Record 생성 시점 지연
#### 절대 허용 불가한 계약 위반
- trace-only라는 이유로 최종 Check/Gate에서 자동 통과 처리
#### 관련 계약 / Rule
- `RC`, `GA`

---

### SC-3.2: 종료 전 일괄 승격 (Late Binding)
#### 맥락
- **전환 여부**: trace-only → record/workflow
#### 사용자의 거부 포인트
- “작업 다 끝났으니 지금까지 대화한 걸 바탕으로 ADR 문서 하나 뽑아줘.”
#### ReproGate 필수 책임
- 누적 Trace에서 Decision / Verification 근거를 추출
- 공식 Record 규격으로 변환
- 원본 Trace를 훼손하지 않고 참조로 연결
- 소급 작성임을 명시하고 provenance link를 요구
#### 허용 가능한 우회
- 자동 초안 생성 후 사용자 수동 보정
#### 절대 허용 불가한 계약 위반
- 소급 생성된 문서가 원본 Trace / Commit / 증빙과 연결되지 않음
#### 관련 계약 / Rule
- `RC`, `DT`, `VT`, `PI`

---

### SC-3.3: Freeform 도중 단발성 Skill 개입
#### 맥락
- **전환 여부**: trace-only → skill → trace-only
#### 사용자의 거부 포인트
- “흐름 안 깨고 보안 검사만 잠깐 하고 계속 대화할래.”
#### ReproGate 필수 책임
- 컨텍스트 스위칭 없이 Skill 실행
- 결과를 현재 Trace 내에 삽입한 후 원래 모드로 복귀
#### 허용 가능한 우회
- Skill 결과를 별도 카드/블록으로 삽입
#### 절대 허용 불가한 계약 위반
- Skill 개입 사실이 Trace에서 보이지 않음
#### 관련 계약 / Rule
- `ST`, `RC`

---

### SC-3.4: Freeform 세션이 길어져 복수 작업이 섞임
#### 맥락
- **시작 모드**: trace-only
- **세션 길이**: 장기
#### 사용자의 거부 포인트
- “한 세션에서 이것저것 같이 처리할래.”
#### ReproGate 필수 책임
- 단일 Trace 안에서 작업 단위 후보를 분리 제안
- 혼재된 Decision / Verification이 잘못 합쳐지지 않게 함
#### 허용 가능한 우회
- 세션 종료 후 수동으로 작업 단위 분리
#### 절대 허용 불가한 계약 위반
- 서로 다른 작업의 근거가 하나의 Record로 섞여 재현성 훼손
#### 관련 계약 / Rule
- `RC`, `DT`, `PI`

---

## 4. Bypass / Out-of-Band

ReproGate의 개입을 의도적으로 끄거나, 외부 환경에서 작업한 후 돌아오는 경우.

### SC-4.1: 일시적 기능 종료 후 복기
#### 맥락
- **시작 모드**: bypass 활성화
#### 사용자의 거부 포인트
- “이 리팩토링은 다른 툴 중심으로 할 거니까 ReproGate는 잠깐 꺼둘래.”
#### ReproGate 필수 책임
- 명시적으로 기능을 끄는 옵션 제공
- 끈 상태에서 발생한 작업 단절(Gap)을 다시 켤 때 인지
- 재가동 시 사후 복기 또는 Gap 승인 경로 제공
#### 허용 가능한 우회
- “이번 구간은 예외 승인” 선택
#### 절대 허용 불가한 계약 위반
- 단절 구간이 명백한데도 아무런 경고·소명 없이 최종 Gate 통과
#### 관련 계약 / Rule
- `GA`

---

### SC-4.2: 외부 환경에서 완료된 작업의 사후 등록
#### 맥락
- **시작 모드**: external
#### 사용자의 거부 포인트
- “외부에서 구현/테스트 끝냈어. 결과만 가져왔으니 바로 푸시할래.”
#### ReproGate 필수 책임
- 과정 Trace가 없음을 인정
- 대신 Decision / Verification의 사후 증빙 작성을 요구
- 소급 설명과 외부 증빙 사이의 provenance 연결 요구
#### 허용 가능한 우회
- 간략 사후 요약 + 테스트 결과 / 커밋 링크 / 로그 첨부
#### 절대 허용 불가한 계약 위반
- 결과물만 있고 판단 근거와 검증 증거가 전혀 없음
#### 관련 계약 / Rule
- `DT`, `VT`, `GA`, `PI`

---

### SC-4.3: Bypass 기간 이후 대량 변경이 발생했지만 설명 거부
#### 맥락
- **시작 모드**: bypass
#### 사용자의 거부 포인트
- “설명 없이 그냥 합칠래.”
#### ReproGate 필수 책임
- Gap 규모를 기반으로 최소 설명 요구 수준 상향
- 대량 변경이면 최소 요약/영향 범위/검증 기록 요구
#### 허용 가능한 우회
- 상세 설명 대신 structured summary 제출
#### 절대 허용 불가한 계약 위반
- 대량 외부 작업 후 근거 없이 단순 승인
#### 관련 계약 / Rule
- `GA`, `DT`, `VT`

---

# 2부. 구조화된 기록과 정책 평가 시나리오

ReproGate의 기록물이 단순 자유서술 메모가 아니라, 정책 평가 가능한 구조를 가져야 한다는 관점의 시나리오들.

---

## 5. 구조화된 문서 계약

### SC-5.1: 자유 본문만 있고 필수 메타데이터가 없음
#### 맥락
- 사용자는 `.md` 본문만 남기고 frontmatter/필수 필드를 생략
#### 사용자의 거부 포인트
- “본문에 다 써놨는데 굳이 필드까지 채워야 해?”
#### ReproGate 필수 책임
- 본문 존재와 구조 계약 충족을 구분해 평가
- 필수 메타데이터 누락을 보완 요청
- 본문을 자동 파싱해 필드 초안 제안 가능
#### 허용 가능한 우회
- 자동 추출된 초안을 사용자 승인 후 저장
#### 절대 허용 불가한 계약 위반
- 구조화 필드가 전혀 없는데 Policy/Gate 평가를 정상 통과
#### 관련 계약 / Rule
- `RC`, `PI`

---

### SC-5.2: Record 본문은 있으나 Decision이 구조화되지 않음
#### 맥락
- 문서에 서술은 있으나 어떤 결론이 채택됐는지 명확하지 않음
#### 사용자의 거부 포인트
- “문맥 보면 다 알 수 있잖아.”
#### ReproGate 필수 책임
- 서술형 문장과 Decision 필드를 구분
- 최종 Decision 후보를 추출해 구조 필드로 승격 제안
#### 허용 가능한 우회
- “결정 없음 / 조사 단계”로 명시하고 임시 저장
#### 절대 허용 불가한 계약 위반
- 최종 구현이 끝났는데도 채택된 결정이 구조상 식별되지 않음
#### 관련 계약 / Rule
- `DT`, `PI`

---

### SC-5.3: Verification은 했지만 증빙 링크/필드가 없음
#### 맥락
- 테스트, 리뷰, 실행은 했다고 주장하나 증빙이 구조화되지 않음
#### 사용자의 거부 포인트
- “테스트 돌렸다는 말로 충분하지 않아?”
#### ReproGate 필수 책임
- 검증 여부와 검증 증빙 존재를 분리 평가
- 로그, 테스트 결과, 링크, 요약 등 최소 증빙 형식 요구
#### 허용 가능한 우회
- 외부 CI 결과 링크 또는 수동 검증 요약 등록
#### 절대 허용 불가한 계약 위반
- 검증했다고만 쓰고 어떤 방식/결과였는지 전혀 남지 않음
#### 관련 계약 / Rule
- `VT`, `PI`

---

### SC-5.4: 구조 필드는 채웠으나 본문과 충돌
#### 맥락
- `verification: true`인데 본문에는 “테스트는 아직 안 함”이 적혀 있음
#### 사용자의 거부 포인트
- “나중에 수정하다 보니 안 맞았을 수도 있지.”
#### ReproGate 필수 책임
- 구조 필드와 본문 간 충돌 감지
- 임의로 한쪽을 정답 처리하지 않고 확인 요구
#### 허용 가능한 우회
- 사용자 컨펌 후 필드/본문 정정
#### 절대 허용 불가한 계약 위반
- 충돌 상태를 무시하고 Gate 평가 진행
#### 관련 계약 / Rule
- `PI`, `VT`, `DT`

---

### SC-5.5: 외부 저장소 문서가 최소 메타데이터 계약을 충족하지 않음
#### 맥락
- Notion/Jira/Confluence 문서가 들어오지만 canonical metadata가 없음
#### 사용자의 거부 포인트
- “우리 툴이 원래 이런 포맷이 아닌데?”
#### ReproGate 필수 책임
- 외부 저장소 사용을 허용하되 최소 메타데이터 계약은 요구
- 어댑터 또는 매핑 계층으로 보완 가능하게 함
#### 허용 가능한 우회
- import 시 필수 필드 매핑 과정 수행
#### 절대 허용 불가한 계약 위반
- 구조 계약이 없는 문서를 그대로 Policy Input으로 사용
#### 관련 계약 / Rule
- `RC`, `PI`, `PV`

---

## 6. 정책 평가 입력과 Gate

### SC-6.1: Built-in Workflow를 쓰지 않았지만 Rule 평가는 받아야 하는 경우
#### 맥락
- 사용자는 custom workflow 또는 freeform만 사용
#### 사용자의 거부 포인트
- “표준 워크플로 안 썼다고 fail시키면 안 되잖아.”
#### ReproGate 필수 책임
- Rule 평가 대상을 workflow 사용 여부가 아니라 canonical policy input으로 삼음
- custom / freeform / external도 동일 입력 계약으로 평가
#### 허용 가능한 우회
- built-in 단계 매핑 없이 상태 필드만으로 평가
#### 절대 허용 불가한 계약 위반
- built-in 사용 여부를 Rule/Gate의 직접 조건으로 삼음
#### 관련 계약 / Rule
- `PI`

---

### SC-6.2: 정책 입력 생성은 가능하지만 일부 필드가 unknown 상태
#### 맥락
- Record는 있으나 일부 항목이 미확정
#### 사용자의 거부 포인트
- “지금 탐색 중인데 왜 벌써 fail이야?”
#### ReproGate 필수 책임
- unknown / pending / satisfied를 구분
- 탐색 단계에서는 hard fail 대신 보류 또는 soft warning 허용
#### 허용 가능한 우회
- 임시 상태 저장 후 후속 확정

- unknown 상태를 satisfied로 암묵 처리
#### 관련 계약 / Rule
- `PI`, `RC`

---

### SC-6.3: Hard Gate 직전 결손 항목 일괄 보완
#### 맥락
- 최종 PR/배포 직전 필수 항목 일부 누락
#### 사용자의 거부 포인트
- “지금 급하니까 마지막에 한 번에 채우면 되지.”

- 누락 계약 항목을 명시
- 자동 보완 가능한 부분과 수동 보완이 필요한 부분 구분

#### 허용 가능한 우회

#### 절대 허용 불가한 계약 위반

#### 관련 계약 / Rule
- `RC`, `DT`, `VT`, `PI`

#### 허용 가능한 우회

#### 절대 허용 불가한 계약 위반

- `PS`, `PV`

# 6부. 포팅 / 이식 / 프로젝트 확장 시나리오

다른 프로젝트에서도 ReproGate를 설치·활용할 수 있도록 이식 가능한 구조를 전제로 한 시나리오.

---

## 12. 프로젝트 간 포팅

### SC-12.1: 프로젝트 A의 Skill Pack을 프로젝트 B로 이식
#### 맥락
- 프로젝트 A에서 검증된 Skill들을 다른 저장소/팀에도 쓰고 싶음
#### 사용자의 거부 포인트
- “이거 좋은데 새 프로젝트에서도 그대로 쓰고 싶어.”
#### ReproGate 필수 책임
- 프로젝트 공통 영역과 프로젝트 특화 영역을 구분
- 이식 시 필요한 변수/경로/전제조건 차이를 명시
#### 허용 가능한 우회
- import 후 project-local override 추가
#### 절대 허용 불가한 계약 위반
- 특정 프로젝트 경로/도메인에 강하게 묶인 Skill이 설명 없이 재사용됨
#### 관련 계약 / Rule
- `PV`, `ST`

---

### SC-12.2: 저장 방식은 다르지만 같은 Rule 세트를 재사용
#### 맥락
- 프로젝트 A는 로컬 Markdown, 프로젝트 B는 외부 문서 시스템 사용
#### 사용자의 거부 포인트
- “저장소는 다르지만 Rule은 똑같이 쓰고 싶어.”
#### ReproGate 필수 책임
- 저장 어댑터와 정책 입력 생성 계층을 분리
- 저장 방식이 달라도 canonical policy input을 동일하게 만들 수 있어야 함
#### 허용 가능한 우회
- 어댑터별 필드 매핑 설정
#### 절대 허용 불가한 계약 위반
- 저장 방식이 다르다는 이유로 동일 Rule 재사용 불가
#### 관련 계약 / Rule
- `PI`, `PV`

---

### SC-12.3: Built-in 없이 Custom Pack만으로 신규 프로젝트 시작
#### 맥락
- 조직이 자체 Workflow/Skill/Storage pack을 이미 보유
#### 사용자의 거부 포인트
- “기본 제공 템플릿은 필요 없고 우리 표준 Pack만 쓸래.”
#### ReproGate 필수 책임
- Built-in 미사용 자체를 오류로 보지 않음
- Custom Pack이 core contract를 충족하는지 검증
#### 허용 가능한 우회
- Built-in은 비활성화하고 validator만 사용
#### 절대 허용 불가한 계약 위반
- Built-in을 안 썼다는 이유만으로 제품 사용 불가 처리
#### 관련 계약 / Rule
- `RC`, `PI`, `PV`

---

### SC-12.4: 프로젝트별 Workflow는 다르지만 공통 Record Contract는 유지해야 함
#### 맥락
- A 프로젝트는 설계 중심, B 프로젝트는 운영 개선 중심
#### 사용자의 거부 포인트
- “프로세스가 다르니 기록 형식도 다 달라야 하지 않아?”
#### ReproGate 필수 책임
- Workflow 차이와 Record Contract 차이를 구분
- 프로젝트별 단계는 달라도 최소 Decision/Verification/Policy Input 구조는 유지
#### 허용 가능한 우회
- 프로젝트별 필드 확장 허용
#### 절대 허용 불가한 계약 위반
- 프로젝트별 차이를 이유로 core metadata 자체가 사라짐
#### 관련 계약 / Rule
- `RC`, `PI`, `PV`

# 부록 B. 시나리오에서 파생된 설계 보강 포인트

이 부록은 “구현 때 떠올리자”가 아니라,  
**시나리오를 실제 시스템으로 내릴 때 반드시 기억해야 하는 보강 포인트를 지금 보존하기 위한 섹션**이다.

---

## D-01. 소급 작성 Record의 Provenance 강제

### 관련 시나리오
- SC-3.2 종료 전 일괄 승격
- SC-4.2 외부 환경 작업의 사후 등록
- SC-6.3 Hard Gate 직전 결손 항목 일괄 보완

### 왜 필요한가
Late Binding 또는 사후 등록 시 AI가 그럴듯한 서사를 생성할 위험이 있다.  
ReproGate가 재현성을 주장하려면, 소급 작성된 결론은 반드시 **원본 흔적**과 연결되어야 한다.

### 확정 방향
- 소급 작성된 Record는 원천 Trace / Commit / Diff / 로그 / 테스트 결과 등과 연결되어야 한다.
- 단순 서술만으로는 충분하지 않다.
- “이 결론이 어디서 나왔는가”를 추적 가능하게 해야 한다.

### 아직 열어둘 사항
- 표준 provenance 링크 형식을 무엇으로 할지
  - Trace ID
  - Commit Hash
  - Diff Ref
  - Artifact Link
- 하나의 링크만으로 충분한지, 최소 2종 근거를 요구할지

---

## D-02. Skill 결과 수동 Override 감지

### 관련 시나리오
- SC-2.4 Skill 결과를 수동 수정으로 덮어쓰기
- SC-2.5 Skill reject 후 수동 경로 진행

### 왜 필요한가
Skill 실행 흔적은 있으나, 최종 결과가 Skill 출력과 크게 다를 수 있다.  
이 차이를 감지하지 못하면 Skill Trace는 남아도 실제 작업 흐름은 왜곡된다.

### 확정 방향
- ReproGate는 Skill 산출물과 후속 결과 간 의미 있는 괴리를 감지할 수 있어야 한다.
- 괴리가 감지되면 `manual override suspected` 또는 이에 준하는 상태를 둘 수 있어야 한다.
- 최종적으로는 사용자 확인 또는 후속 Decision Trace로 확정한다.

### 아직 열어둘 사항
- 감지 수단
  - 파일 스냅샷 비교
  - Commit Diff
  - Patch 비교
  - Hook
  - Editor Integration
- 임계치 판단 기준
- 자동 확정이 아닌 의심 상태와 확정 상태를 분리할지 여부

---

## D-03. 개인 Skill의 팀 승격 상태 모델

### 관련 시나리오
- SC-11.1 개인 Skill을 팀 표준으로 승격 요청

### 왜 필요한가
개인 실험용 Skill과 팀 표준 Skill 사이에는 검토·승인 단계가 필요하다.  
이 상태 모델이 없으면 거버넌스가 없는 상태에서 표준이 오염될 수 있다.

### 확정 방향
- Skill에는 최소한 다음과 같은 승격 상태가 필요하다.
  - `local`
  - `proposed`
  - `team-approved`
- `proposed` 상태에서는 다른 사용자가 opt-in으로 시험 사용 가능해야 한다.

### 아직 열어둘 사항
- `rejected`, `deprecated`, `archived` 등의 추가 상태 필요 여부
- 팀 승인 프로세스를 문서 기반으로 할지 시스템 UI로 둘지
- 팀 / 조직 범위의 차등 승인 모델 필요 여부

---

## D-04. Policy Input Ready의 단계적 판정

### 관련 시나리오
- SC-5.1 ~ SC-6.4 전체

### 왜 필요한가
정책 평가 가능 상태가 단순 true/false면 탐색 단계와 완료 단계를 구분하기 어렵다.

### 확정 방향
- Policy Input 상태는 최소한 다음을 구분할 수 있어야 한다.
  - `missing`
  - `partial`
  - `ready`
  - `conflicted`
- unknown 상태를 자동 만족으로 간주해서는 안 된다.

### 아직 열어둘 사항
- 각 상태별 soft/hard gate 기준
- partial 상태에서 허용되는 예외 범위

---

## D-05. Built-in과 Custom의 우선순위·식별성 보장

### 관련 시나리오
- SC-10.1, SC-10.2, SC-10.3
- SC-8.4, SC-9.3

### 왜 필요한가
이름은 같지만 실제 내용이 다른 Skill/Workflow/Rule이 생기면 재현성이 깨진다.

### 확정 방향
- scope / namespace / precedence를 명시할 수 있어야 한다.
- 최종적으로 어느 객체가 적용됐는지 사용자가 추적 가능해야 한다.

### 아직 열어둘 사항
- naming rule
- namespace 형식
- override vs clone의 기본 전략

---

## D-06. 외부 저장소 사용 시 Canonical Metadata Layer 필요

### 관련 시나리오
- SC-5.5
- SC-7.2
- SC-7.3
- SC-12.2

### 왜 필요한가
저장 위치가 달라도 동일한 Rule과 Gate를 재사용하려면 내부적으로 공통 해석 계층이 있어야 한다.

### 확정 방향
- Storage Adapter와 Policy Input 생성 계층은 분리되어야 한다.
- 외부 저장소는 그대로 두되, ReproGate는 canonical metadata로 변환해 판단한다.

### 아직 열어둘 사항
- adapter spec 범위
- source-of-truth 충돌 시 사용자 확인 플로우
- sync 여부와 read-only integration 여부

# 부록 C. 후속 설계 문서로 연결해야 할 항목

이 시나리오 카탈로그는 다음 설계 문서들의 입력으로 사용된다.

## C-1. Record Contract 설계
- 필수 메타데이터
- Decision / Verification / Provenance 최소 구조
- Policy Input Ready의 기준

## C-2. Rule / Gate 설계
- advisory / soft / hard gate 구조
- unknown / partial / ready / conflicted 판정
- built-in evaluator와 external policy backend의 관계

## C-3. Skill / Workflow Object Model
- clone / fork / override / versioning
- Atomic Skill과 Workflow 구분 기준
- 상태 모델(local / proposed / approved 등)

## C-4. Storage Adapter 설계
- canonical metadata layer
- source-of-truth 충돌 처리
- external storage import / sync / read-only integration

## C-5. Governance / Scope 설계
- local / team / org 범위
- 승인 / 승격 / 폐기 lifecycle
- 권한과 hard gate 연결 조건

## C-6. Portability 설계
- project-agnostic core contract
- pack import/export
- compatibility / migration 정책

# 부록 D. 이 문서의 사용 원칙

이 문서는 단순 참조 문서가 아니라 다음과 같이 사용한다.

1. **새 기능 추가 전**
   - 어떤 시나리오를 만족시키려는지 먼저 연결한다.

2. **설계 리뷰 시**
   - 해당 기능이 어떤 계약 태그(`RC`, `DT`, `VT`, `PI` 등)를 건드리는지 확인한다.

3. **구현 시**
   - 본문 시나리오만 보지 말고 부록 B의 보강 포인트까지 함께 검토한다.

4. **문서 개정 시**
   - 새 시나리오를 추가할 때는
     - 사용자 거부 포인트
     - 필수 책임
     - 허용 가능한 우회
     - 절대 허용 불가한 위반
     - 관련 계약 / Rule
     를 같은 형식으로 유지한다.
